<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AddTransactionPage → POST /transactions | User-provided form data (amount, description, date, type, category_id) sent to backend. Backend validates via `CreateTransactionSchema` (Zod). |
| CategorizePage → PATCH /transactions/:id/category | Selected transaction IDs + category_id sent via parallel PATCH calls. Backend validates via `AssignCategorySchema` and immutability trigger. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-15 | Tampering | AddTransactionPage amount field | mitigate | HTML5 `<input type="number" step="0.01" min="0.01">` provides client-side validation. `toFixed(4)` ensures 4-decimal format. Backend Zod validates `amount: z.string().regex(/^\d+(\.\d{1,4})?$/)`. Defense in depth. |
| T-03-16 | Elevation of Privilege | CategorizePage bulk PATCH | mitigate | `assignCategory()` calls go through `requireAuth` middleware. `AssignCategorySchema` validates `category_id` as UUID. Database trigger only allows category-only updates from NULL. Cannot escalate to modify other fields. |
| T-03-17 | Tampering | CategorizePage transaction IDs | mitigate | Transaction IDs come from the API response (server-generated UUIDs). User cannot inject arbitrary IDs — they can only select from the loaded list. PATCH uses `WHERE id = ${id} AND category_id IS NULL` — only affects uncategorized rows. |
| T-03-18 | Information Disclosure | AddTransactionPage error display | accept | Error messages from API envelope are displayed as-is (user-friendly messages). No stack traces leaked. React auto-escapes JSX. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan. |
</threat_model>
