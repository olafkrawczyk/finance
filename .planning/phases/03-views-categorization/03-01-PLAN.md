## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | PATCH /transactions/:id/category | mitigate | `AssignCategorySchema` with `z.uuid()` validates category_id format at route entry via `zValidator`. Non-UUID input returns 400. |
| T-03-02 | Spoofing | PATCH /transactions/:id/category | mitigate | `requireAuth` middleware on route — validates Better Auth session cookie before handler executes. Unauthenticated requests return 401. |
| T-03-03 | Elevation of Privilege | SQL UPDATE in PATCH handler | mitigate | `WHERE id = ${id} AND category_id IS NULL` — only uncategorized transactions can be updated. Already-categorized transactions return 404. Immutability trigger provides defense-in-depth (blocks UPDATE if any field other than category_id changes). |
| T-03-04 | Information Disclosure | PATCH error responses | mitigate | Error responses follow envelope pattern `{ data: null, error: { message }, meta: null }` — generic messages only, no stack traces or SQL internals leaked. |
| T-03-05 | Tampering | Immutability trigger bypass | mitigate | Updated trigger uses explicit field comparison (`OLD.amount = NEW.amount` etc.) — any attempt to change other fields alongside category_id raises exception. Only the exact NULL→non-null category assignment path is allowed. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan. |
</threat_model>
