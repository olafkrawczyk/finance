<objective>
Stand up the project's runtime foundation: install Bun, provision Postgres with the PGMQ extension via Docker, and create the database layer (postgres.js client, schema DDL, category seed, PGMQ queue init, health check). This is the Walking Skeleton for Phase 1 — every later plan depends on a running DB with the correct schema.

Purpose: Without Bun installed and a PGMQ-enabled Postgres running, no other Phase 1 task can execute. This plan removes both environment blockers and lays down the authoritative schema (D-01 through D-08).

Output: Installed Bun, running Postgres+PGMQ container, tsconfig.json, postgres.js client, schema.sql, seed.sql, apply script, health.ts, and discoverable test scaffolds.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer shell -> install scripts | `curl | bash` Bun installer and Docker image pulls run with developer privileges |
| application -> Postgres | DB credentials cross from env into the postgres.js client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | schema.sql / seed.sql via sql.unsafe | mitigate | sql.unsafe is used ONLY on trusted static DDL files bundled in the repo (apply.ts), never on request input; all runtime queries use tagged templates |
| T-01-02 | Information Disclosure | DB credentials | mitigate | Credentials come from DATABASE_URL env var; .env is git-ignored, only .env.example is committed (no real secrets) |
| T-01-03 | Tampering | transactions immutability | mitigate | BEFORE UPDATE/DELETE triggers raise an exception at the DB level — no app code can bypass (REQ-1.2) |
| T-01-SC | Tampering | bun installer + docker image + npm installs | mitigate | Packages pre-audited in RESEARCH Package Legitimacy Audit (all [OK], no postinstall scripts); pgmq-js removed; Docker image pinned to `ghcr.io/pgmq/pg18-pgmq:v1.10.0` |
</threat_model>
