    **Route 1: GET /insights** (list insights, per D-07)
    - Middleware: requireAuth, zValidator('query', ListInsightsQuerySchema) with error callback returning 400 + { data: null, error: { message: 'Validation failed', details: result.error.flatten() }, meta: null }
    - Handler: extract query from c.req.valid('query'), extract user from c.get('user'), call listInsights({ userId: user.id, ...query })
    - Response: 200 + { data: rows, error: null, meta: { total, page: query.page, per_page: query.per_page } }
    - Error catch: 500 + { data: null, error: { message }, meta: null }
    - Per D-04 filtering: dismissed=false by default (frontend passes it, but default in query schema handles it)

    **Route 2: PATCH /insights/:id/dismiss** (dismiss insight, per D-04)
    - Middleware: requireAuth, zValidator('json', DismissInsightSchema) with 400 error callback
    - Handler: validate :id param with z.uuid().safeParse() (matching ledger PATCH pattern at line 97), call dismissInsight(id, user.id)
    - If result is null: 404 + { data: null, error: { message: 'Insight not found' }, meta: null }
    - If success: 200 + { data: updated, error: null, meta: null }
    - Error catch: 500 + structured error

    **Route 3: POST /insights/generate** (on-demand generation, per D-05)
    - Middleware: requireAuth, zValidator('json', GenerateInsightsSchema) with 400 error callback
    - Handler: call enqueueAnalysisJob(user.id)
    - Response: 202 + { data: { msg_id }, error: null, meta: { message: 'Analysis job enqueued' } }
    - Error catch: 500 + structured error

    **Route 4: GET /insights/forecast** (forecast data for chart, per D-15/D-16)
    - Middleware: requireAuth only (no body/query validation needed)
    - Handler: fetch forecast-type insights for the current user: query insights table WHERE type='forecast' AND user_id=$userId AND dismissed=false ORDER BY created_at DESC LIMIT 10
    - Response: 200 + { data: forecasts, error: null, meta: { total: forecasts.length } }
    - Error catch: 500 + structured error

    **Route 5: GET /insights/dashboard** (dashboard widget data, per D-08)
    - Middleware: requireAuth only
    - Handler: call getInsightsForDashboard(user.id, 3)
    - Response: 200 + { data: rows, error: null, meta: { total: rows.length } }
    - Error catch: 500 + structured error

    All routes must use requireAuth as first middleware (matching ledger.ts pattern). All success responses use `c.json({ data, error: null, meta }, status)` shape. All errors use `c.json({ data: null, error: { message }, meta: null }, status)` shape.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client→API | All /insights routes receive untrusted HTTP requests. Authentication checked via requireAuth (session cookie). Query params and JSON bodies validated with Zod. |
| API→DB | Insight queries filter by user_id from session. Dismiss only affects user's own insights. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-14 | Spoofing | GET /insights without auth | mitigate | requireAuth middleware returns 401 before any DB query. Copy existing auth pattern from ledger.ts. |
| T-04-15 | Tampering | PATCH /insights/:id/dismiss with forged id | mitigate | UUID validation on :id param (z.uuid().safeParse). dismissInsight() includes user_id in WHERE clause — cannot dismiss another user's insights. |
| T-04-16 | Information Disclosure | Error responses leaking DB details | mitigate | All error handlers catch Exception → message only (no stack). Standard envelope: { data: null, error: { message }, meta: null }. Never include raw DB errors. |
| T-04-17 | Denial of Service | POST /insights/generate flood | mitigate | Route itself does not enforce rate limit (that's in the worker). However, the worker's RATE_LIMIT_COOLDOWN_MS enforcement means rapid POST calls will enqueue but the worker will skip redundant messages. |
| T-04-18 | Elevation of Privilege | Dismissing another user's insight | mitigate | dismissInsight(id, userId) uses both id AND userId in WHERE clause. Session userId from requireAuth middleware cannot be forged (Better Auth signed session). |
| T-04-19 | Information Disclosure | GET /insights returns another user's insights | mitigate | listInsights() includes WHERE user_id = $userId from session. Every query is user-scoped. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages required. All dependencies already installed and vetted. |
</threat_model>
