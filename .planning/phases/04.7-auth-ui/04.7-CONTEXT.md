# Phase 4.7: Auth UI (Login/Logout/Guard) - Context

**Gathered:** 2026-06-07
**Source:** Exploration Todos + Phase 2 Auth Foundation

<domain>
## Phase Boundary

Build the frontend authentication UI on top of Phase 2's existing Better Auth backend. The app currently has NO auth UI — the main page renders freely without session checks. This phase adds:

1. **Login/Signup page** (`/login`) — tabbed single page with email/password forms + Google OAuth button
2. **Auth guard** — session check on mount, redirect to `/login` if unauthenticated, loading spinner
3. **Logout button** — in header, calls sign-out, clears session, redirects to `/login`
</domain>

<decisions>
## Implementation Decisions

### Existing Auth Backend (Phase 2 — DO NOT TOUCH)
- `src/auth.ts`: Better Auth config with email/password + Google/GitHub OAuth, Postgres pool, session management
- `src/interface-adapters/api/auth.ts`: `requireAuth` middleware — checks session via `auth.api.getSession()`, returns 401
- `index.ts`: Better Auth handler mounted at `/api/auth/*`, session middleware on all routes
- All backend API routes (Phase 1-4.5) use `credentials: 'include'` for cookie-based sessions

### Available Auth Endpoints (already implemented)
- `POST /api/auth/sign-up/email` — sign up with email + password
- `POST /api/auth/sign-in/email` — sign in with email + password
- `POST /api/auth/sign-in/social` — social (Google/GitHub) OAuth
- `POST /api/auth/sign-out` — sign out, clear session
- `GET /api/auth/get-session` — check current session (returns user + session or null)

### Login/Signup Page
- Single page at `/login` with two tabs: "Zaloguj" (login) and "Rejestracja" (signup)
- Email + password form in each tab
- "Kontynuuj przez Google" button for Google OAuth (always visible)
- On success, redirect to `/dashboard`
- Use `POST /api/auth/sign-in/email`, `POST /api/auth/sign-up/email`, `POST /api/auth/sign-in/social` (Google)
- `credentials: 'include'` for cookie-based sessions

### Auth Guard
- Check session on mount via `GET /api/auth/get-session`
- Add session state (`user | null`, `loading`) at App level
- Show loading spinner while session is being checked
- If no session, render only the login page (don't show main nav/content)
- If session exists, render current routing logic (unchanged)
- Handle 401 responses gracefully across all API calls

### Logout Button
- `POST /api/auth/sign-out` with `credentials: 'include'`
- Clear session state and redirect to `/login`
- Position in header next to existing nav items (right-aligned)
- Show user email/name next to the button

### the agent's Discretion
- Styling details (tailwind classes, colors) — follow existing app theme (slate-950 bg, slate-100 text)
- Loading spinner design — use existing spinner pattern if one exists, else create simple CSS spinner
- Error display for login/signup failures — `the agent's Discretion` on toast vs inline vs alert
- Form validation — inline field validation vs submit-only validation
- OAuth redirect flow — popup vs full-page redirect
- Whether to restructure App.tsx or create a wrapper AuthLayout component

</decisions>
