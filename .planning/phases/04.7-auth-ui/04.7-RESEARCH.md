## Summary

This phase builds a frontend authentication layer on top of Phase 2's existing Better Auth backend. The app currently renders freely without session checks — all pages are accessible. The core work is: (1) create a login/signup page, (2) add a session-aware auth guard at the App component level, and (3) add a logout button with user info in the header.

**Critical finding:** Better Auth ships a first-class React client (`better-auth/react`) that provides `useSession()` — a reactive session hook using nanostores + React `useSyncExternalStore`. This is the standard approach. It eliminates the need to manually call `GET /api/auth/get-session` — the client manages session lifecycle, automatic refresh, and broadcast across tabs.

**No new packages required.** `better-auth` is already at 1.6.14 in `package.json`. The React client is a subpath export (`better-auth/react`) from the same package.

**The existing spinner pattern** (`animate-spin rounded-full h-10 w-10 border-b-2 border-blue-500`) appears in 12+ places across the frontend — reuse it for the auth guard loading state.

**Primary recommendation:** Import `{ createAuthClient }` from `better-auth/react` in a new `frontend/src/lib/auth-client.ts` file. Use `authClient.useSession()` at App level for the auth guard. Use `authClient.signIn.email()`, `authClient.signUp.email()`, and `authClient.signIn.social()` in a LoginPage component. Use `authClient.signOut()` for logout.

### Pattern 1: Better Auth React Client Setup

**What:** Create a single auth client instance with `createAuthClient` from `better-auth/react`. All auth operations use this instance.

**Source:** [VERIFIED: better-auth.com/docs/basic-usage — createAuthClient, useSession]

```typescript
// frontend/src/lib/auth-client.ts
import { createAuthClient } from "better-auth/react";

export const authClient = createAuthClient({
  // baseURL defaults to '/api/auth' — matches our Vite proxy
  // credentials: 'include' is set by default
});
```

**Key: No `baseURL` override needed.** The client defaults to `/api/auth` which matches the Vite proxy config (`/api/auth` → `http://localhost:3000`). `credentials: 'include'` is set automatically by the Better Auth client when the browser supports it (all modern browsers do).

### Pattern 2: Auth Guard in App.tsx

**What:** Use `authClient.useSession()` at the App level to guard content. This is a React hook that returns `{ data, isPending, error, refetch }`.

**Source:** [VERIFIED: better-auth.com/docs/basic-usage — useSession hook returns { data: { session, user } | null, isPending, error, refetch }]

```typescript
// App.tsx
import { authClient } from './lib/auth-client';

export default function App() {
  const { data: session, isPending } = authClient.useSession();
  const [currentPath, setCurrentPath] = useState(window.location.pathname);

  // ... (existing navigation logic unchanged) ...

  // Auth guard loading state
  if (isPending) {
    return (
      <div className="min-h-screen bg-slate-950 flex items-center justify-center">
        <div className="animate-spin rounded-full h-10 w-10 border-b-2 border-blue-500" />
      </div>
    );
  }

  // No session — render only login page
  if (!session) {
    return (
      <div className="min-h-screen bg-slate-950 flex items-center justify-center">
        <LoginPage onLogin={() => { /* session atom auto-updates */ }} />
      </div>
    );
  }

  // Session exists — render normal content with header
  return (
    <div className="min-h-screen bg-slate-950 text-slate-100 flex flex-col">
      <header className="border-b border-slate-900 bg-slate-950/60 backdrop-blur-md sticky top-0 z-50">
        <div className="max-w-6xl mx-auto px-6 py-4 flex justify-between items-center">
          {/* Logo and title (unchanged) */}
          <div className="flex items-center space-x-3">
            <div className="w-9 h-9 rounded-xl bg-gradient-to-tr from-blue-600 to-indigo-600 flex items-center justify-center font-bold text-white shadow-lg shadow-blue-500/20">F</div>
            <span className="text-lg font-semibold tracking-tight text-white">Finance<span className="text-blue-500">Flow</span></span>
          </div>

          <nav className="flex flex-wrap space-x-1">
            {/* existing nav buttons — unchanged */}
          </nav>

          {/* NEW: User info + logout */}
          <div className="flex items-center space-x-3 ml-4">
            <span className="text-sm text-slate-400">{session.user.email}</span>
            <button
              onClick={async () => {
                await authClient.signOut();
                // session atom auto-updates, component re-renders to login page
              }}
              className="text-sm text-slate-500 hover:text-slate-300 transition-colors"
            >
              Wyloguj
            </button>
          </div>
        </div>
      </header>

      <main className="flex-grow px-6 py-12 max-w-6xl mx-auto w-full">
        {renderContent()}
      </main>

      <footer>{/* unchanged */}</footer>
    </div>
  );
}
```

**How session reactivity works:**
1. `useSession()` subscribes to a nanostores atom managed by Better Auth
2. On mount, it triggers `GET /api/auth/get-session` with the cookie
3. When `signOut()` or `signIn` completes, the atom updates automatically
4. React re-renders due to `useSyncExternalStore` (via nanostores React integration)
5. No manual state management needed — the toast/post-signin redirect is optional

### Pattern 3: LoginPage with Tabs

**What:** Tabbed login/signup page with email/password forms and Google OAuth button.

```typescript
// frontend/src/pages/LoginPage.tsx
import { useState } from 'react';
import { authClient } from '../lib/auth-client';

export default function LoginPage() {
  const [tab, setTab] = useState<'login' | 'register'>('login');
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  const [name, setName] = useState('');
  const [error, setError] = useState<string | null>(null);
  const [loading, setLoading] = useState(false);

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    setError(null);
    setLoading(true);

    try {
      if (tab === 'login') {
        const { error } = await authClient.signIn.email({ email, password });
        if (error) throw new Error(error.message || 'Login failed');
        // Session atom auto-updates; App.tsx re-renders to dashboard
        // Redirect to dashboard if not already there
        if (window.location.pathname === '/login') {
          window.history.pushState(null, '', '/dashboard');
          // Trigger popstate so App.tsx picks up the change (or use navigateTo)
        }
      } else {
        const { error } = await authClient.signUp.email({
          email,
          password,
          name,
        });
        if (error) throw new Error(error.message || 'Sign up failed');
        // Auto-signed in, redirect to dashboard
        if (window.location.pathname === '/login') {
          window.history.pushState(null, '', '/dashboard');
        }
      }
    } catch (err) {
      setError(err instanceof Error ? err.message : 'An error occurred');
    } finally {
      setLoading(false);
    }
  };

  const handleGoogleLogin = async () => {
    setLoading(true);
    try {
      await authClient.signIn.social({
        provider: 'google',
        callbackURL: '/dashboard',
      });
      // The redirect plugin in better-auth automatically redirects the browser
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Google login failed');
      setLoading(false);
    }
  };

  return (
    <div className="w-full max-w-md mx-auto px-6">
      {/* Logo */}
      <div className="text-center mb-8">
        <div className="w-12 h-12 rounded-xl bg-gradient-to-tr from-blue-600 to-indigo-600 flex items-center justify-center font-bold text-white shadow-lg shadow-blue-500/20 mx-auto mb-4">
          F
        </div>
        <h1 className="text-2xl font-bold text-white">FinanceFlow</h1>
        <p className="text-slate-400 text-sm mt-1">Zarządzaj swoimi finansami</p>
      </div>

      {/* Tabs */}
      <div className="flex bg-slate-900 rounded-xl p-1 mb-6">
        <button
          onClick={() => setTab('login')}
          className={`flex-1 py-2 rounded-lg text-sm font-semibold transition-colors ${
            tab === 'login' ? 'bg-slate-800 text-white' : 'text-slate-400 hover:text-slate-200'
          }`}
        >
          Zaloguj
        </button>
        <button
          onClick={() => setTab('register')}
          className={`flex-1 py-2 rounded-lg text-sm font-semibold transition-colors ${
            tab === 'register' ? 'bg-slate-800 text-white' : 'text-slate-400 hover:text-slate-200'
          }`}
        >
          Rejestracja
        </button>
      </div>

      {/* Error */}
      {error && (
        <div className="bg-red-950/50 border border-red-800 rounded-lg p-3 text-sm text-red-300 mb-4">
          {error}
        </div>
      )}

      {/* Form */}
      <form onSubmit={handleSubmit} className="space-y-4">
        {tab === 'register' && (
          <input
            type="text"
            placeholder="Imię i nazwisko"
            value={name}
            onChange={(e) => setName(e.target.value)}
            required
            className="w-full px-4 py-2.5 bg-slate-900 border border-slate-800 rounded-lg text-slate-100 placeholder-slate-500 focus:outline-none focus:ring-2 focus:ring-blue-500"

          className="w-full px-4 py-2.5 bg-slate-900 border border-slate-800 rounded-lg text-slate-100 placeholder-slate-500 focus:outline-none focus:ring-2 focus:ring-blue-500"
```

### Pattern 4: Client-Side Navigation Redirect After Login

**What:** After sign-in/sign-up succeeds, the session atom updates automatically, and `useSession()` returns the session. But the user still needs to navigate to `/dashboard`. Since the app uses `window.history.pushState` (no React Router), we need to manually redirect.

```typescript
// Helper: navigate to a path and trigger React re-render
function navigateTo(path: string) {
  window.history.pushState(null, '', path);
  window.dispatchEvent(new Event('popstate'));
}
```

**Better approach:** Since App.tsx already has a `navigateTo` function and listens for `popstate`, call it as a callback after successful login:

```typescript
// In LoginPage, after successful sign-in:
authClient.signIn.email(
  { email, password },
  {
    onSuccess: () => {
      // navigateTo('/dashboard') — but we need access to the function
      window.history.pushState(null, '', '/dashboard');
      window.dispatchEvent(new Event('popstate'));
    },
  }
);
```

Or simpler — just check if the user is on `/login` after the session changes and redirect in App.tsx via a `useEffect`.

### Anti-Patterns to Avoid
- **Storing session in localStorage:** Better Auth cookies are HTTP-only and Secure. Do not extract the session token and store it in localStorage — defeats cookie security.
- **Calling `getSession()` manually:** The `useSession()` hook handles this automatically. Manual calls create race conditions.
- **Wrapping the client in React Context:** The Better Auth client uses nanostores (external store) + `useSyncExternalStore`. Wrapping it in React Context adds unnecessary re-render overhead.
- **Checking session in each page component:** Check session once at App level. Individual pages should not re-fetch session.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session lifecycle management | Manual `fetch('/api/auth/get-session')` + React state | `useSession()` from `better-auth/react` | Auto-refresh, cross-tab sync, error handling, loading state |
| Sign-in/sign-up API calls | Manual `fetch` with JSON body parsing | `authClient.signIn.email()`, `authClient.signUp.email()` | Normalized error codes, cookie handling, automatic redirect for social |
| OAuth redirect flow | Window.open + popup listener | `authClient.signIn.social()` with `callbackURL` | Built-in redirect plugin handles the OAuth URL, PKCE is built into Better Auth |
| Reactive state for session | React Context + useState + useEffect | nanostores atom + `useSyncExternalStore` | Zero-extra-dependency state management built into Better Auth client |
| Logout + session cleanup | Manual cookie deletion + state clear | `authClient.signOut()` | Clears server session, updates client atom, broadcasts to other tabs |

**Key insight:** The Better Auth React client is specifically designed to eliminate all the boilerplate of auth state management. The `useSession()` hook is the canonical approach — it provides `{ data, isPending, error, refetch }` matching React Query patterns exactly, but without needing TanStack Query. Everything is reactive via nanostores.

## Common Pitfalls

### Pitfall 1: useSession() Returns `isPending` = true on Initial Render

**What goes wrong:** The auth guard shows a loading spinner on every page navigation, not just initial load.

**Why it happens:** `useSession()` triggers `GET /api/auth/get-session` on mount. Every mount of the App component triggers this call. The `isPending` flag is `true` while the request is in-flight.

**How to avoid:** Structure the App component so the loading state only shows on initial load (not page navigations). Since App.tsx mounts once and stays mounted (React SPA), this is not actually an issue — `isPending` only flips to `true` once on first render and stays `false` after that. The session is cached in the nanostores atom across re-renders.

**Warning signs:** Flash of loading spinner on every route change.

### Pitfall 2: OAuth Redirect Leaves `isPending` = true Forever

**What goes wrong:** After returning from Google OAuth, the auth guard remains stuck on the loading spinner.

**Why it happens:** The Better Auth redirect plugin sets `window.location.href` to the OAuth provider. When the user returns, the React app remounts completely, and the session cookie is now present. `useSession()` should resolve immediately. But if the session isn't persisted yet (race condition), `isPending` stays `true`.

**How to avoid:** This is handled by Better Auth's session refresh mechanism — the `createSessionRefreshManager` in the client polls for session updates after redirects. On the login page mount, also check if already authenticated (via a quick `useEffect`):

```typescript
// In LoginPage — redirect if already logged in
const { data: session, isPending } = authClient.useSession();
useEffect(() => {
  if (session) {
    window.history.pushState(null, '', '/dashboard');
    window.dispatchEvent(new Event('popstate'));
  }
}, [session]);
```

**Warning signs:** Loading spinner persists after OAuth callback redirect.

### Pitfall 3: Social Sign-In Callback URL Mismatch

**What goes wrong:** After Google OAuth authorization, the redirect URL doesn't match any route the app handles, causing a 404.

**Why it happens:** Better Auth's OAuth flow requires `BETTER_AUTH_URL` to match the actual server URL. If `BETTER_AUTH_URL=http://localhost:3000` but the frontend is at `http://localhost:5173`, the OAuth callback URL could cause a mismatch.

**How to avoid:** This is already configured correctly — `BETTER_AUTH_URL=http://localhost:3000` matches the backend where the Better Auth handler lives. The OAuth callback goes to `http://localhost:3000/api/auth/callback/*` which is handled by Better Auth. After the callback, Better Auth sets the session cookie and redirects the user to `callbackURL` (`/dashboard`). The SPA at `localhost:5173` then handles the `/dashboard` route.

**BUT:** The Vite proxy's `spaBypass` function must handle this correctly. Ensure the `/api/auth/callback/*` path is properly proxied (it is — the proxy for `/api/auth` uses `spaBypass` only for non-POST/GET requests that contain `text/html`, which OAuth redirects should not match since they're JSON API calls).

**Warning signs:** "404 - Page Not Found" after OAuth login.

### Pitfall 4: Form Validation — Trust Client-Side Validation Only

**What goes wrong:** User submits an invalid email or weak password, the frontend doesn't validate, and the Better Auth API returns a hard-to-parse error.

**Why it happens:** Better Auth's built-in validation includes password length (min 8 chars), email format, and required fields. The error responses are structured but may require parsing.

**How to avoid:** Add basic HTML5 validation (`required`, `minLength={8}`, `type="email"`) AND handle the structured error response from Better Auth:

```typescript
const { error } = await authClient.signIn.email({ email, password });
if (error) {
  // error.status, error.message, error.code
  setError(error.message || 'Invalid credentials');
}
```

**Warning signs:** Unhandled promise rejections from Better Auth API errors.

### Pitfall 5: Sign-Up Auto-Sign-In Creates Session Without Redirect

**What goes wrong:** After sign-up, the user is auto-signed-in (Better Auth default), but the UI doesn't navigate to `/dashboard`.

**Why it happens:** `useSession()` picks up the new session reactively, but the SPA router doesn't automatically navigate. The auth guard stops showing the login page, but the user sees whatever route they were on (`/login`).

**How to avoid:** After a successful sign-up, redirect from the login page itself. Use the `onSuccess` callback or a `useEffect` that watches `session`:

```typescript
useEffect(() => {
  if (session && window.location.pathname === '/login') {
    window.dispatchEvent(new Event('popstate'));
    window.history.pushState(null, '', '/dashboard');
  }
}, [session]);
```

### Pitfall 6: Social Sign-In Redirect URL Route Handling

**What goes wrong:** After OAuth, Better Auth redirects to the callback URL (e.g. `/dashboard`). For the Vite SPA, this URL must be handled correctly — non-API routes should serve `index.html`.

**How to avoid:** The `spaBypass` function in `vite.config.ts` already handles this for `/api/auth/*` routes. Ensure `/dashboard` and other frontend routes are also handled. If using a single-page app, the Vite dev server should serve `index.html` for all non-matching routes. Verify:

```
Server port 5173:
  GET /dashboard → SPA fallback → index.html → React mounts → navigates to /dashboard
```

**Warning signs:** Blank page or file listing after OAuth redirect.

```typescript
export default function App() {
  const { data: session, isPending } = authClient.useSession();
  const [currentPath, setCurrentPath] = useState(window.location.pathname);

  useEffect(() => {
    const handleLocationChange = () => setCurrentPath(window.location.pathname);
    window.addEventListener('popstate', handleLocationChange);
    return () => window.removeEventListener('popstate', handleLocationChange);

    window.history.pushState(null, '', path);

    return (
      <div className="min-h-screen bg-slate-950 flex items-center justify-center">
        <LoginPage onSuccess={() => navigateTo('/dashboard')} />
      </div>
    );
```
