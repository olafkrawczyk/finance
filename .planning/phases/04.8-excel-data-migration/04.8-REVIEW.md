# Phase 04.8: Code Review Report

**Reviewed:** 2026-06-07T08:17:55Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

This phase adds a destructive Excel-to-database migration pipeline: a REST endpoint that accepts a workbook upload, atomically truncates the ledger tables and enqueues a PGMQ job, a worker function (`processExcelMigrationJob`) that parses the workbook and performs batched inserts, and a frontend wizard (hidden `/migration` route) with a "type MIGRATE" destructive-confirmation modal and progress polling.

The core upload-and-truncate transaction in `migration.ts` is well designed — the `TRUNCATE` and the `pgmq.send` enqueue happen in the same `sql.begin`, so a failure to enqueue rolls back the destructive wipe. The Excel parsing logic (`excel-parser.ts`) is well factored and unit-tested.

However, several defects undermine the safety guarantees this feature is supposed to provide for a "highly destructive, cannot be undone" operation:

1. The cleanup-on-completion logic (`finally { unlink(file_path) }`) combined with the existing PGMQ retry loop means **every retry of a failed `excel_migration` job is guaranteed to fail** with a file-not-found error — the retry mechanism the worker advertises (`MAX_RETRIES = 3`) is non-functional for this job type, and the user is left with a wiped database and no working recovery path via the queue.
2. The success/failure messaging shown to the user is materially incorrect: the UI claims "Database transaction has been rolled back" on failure, but the implementation commits in batches of 100 rows per sheet — a mid-migration crash leaves committed partial data in the (already-wiped) database, contradicting the rollback claim.
3. A workbook that yields zero parseable sheets/rows is reported as `'completed'` (not `'failed'`), so the UI shows "Migration Completed... Existing data was wiped and replaced with historical ledger records" even though nothing was imported and the original data is already gone.
4. The uploaded file can be orphaned on disk if the transaction that creates the PGMQ message fails after the file was written (`Bun.write` happens before `sql.begin`), since cleanup is solely the worker's responsibility and the worker never runs without a queue message.

## Critical Issues

### CR-01: Excel migration retry is broken — every retry fails with file-not-found

**File:** `src/workers/import-worker.ts:340-472` (interacts with `615-663`)
**Issue:** `processExcelMigrationJob`'s `finally` block unconditionally `unlink`s `file_path` (line 465), regardless of whether the job succeeded or failed:

```ts
} finally {
  // Always cleanup the uploaded workbook, even on failure (T-MIG-05)
  try {
    await unlink(file_path);
  } ...
}
```

If the job fails on its first attempt with `processed === 0` (e.g., the seeded `ING_BUSINESS_ACCOUNT_NAME`/`PKO_PERSONAL_ACCOUNT_NAME` lookup throws, the workbook is corrupted and `XLSX.read` throws, or the very first batch's INSERT errors out), `workerLoop` (lines 636-657) sees `errors.length > 0 && processed === 0`, throws `'All batches failed'`, and — because `read_ct + 1 < MAX_RETRIES` — schedules a retry by leaving the message on the queue with `console.log('[import] Will retry after ${VISIBILITY_TIMEOUT}s')`.

On the retry, `processExcelMigrationJob` runs again with the *same* `file_path`, but the file was already deleted by the previous attempt's `finally` block. `Bun.file(file_path).arrayBuffer()` (line 366) throws `ENOENT`, which is caught by the outer `catch` and recorded as a new fatal error (`Fatal: ENOENT: no such file or directory...`). This repeats until `MAX_RETRIES` is exhausted, at which point the job is marked `failed` with `'Max retries exceeded'`.

Net effect: for any first-attempt failure, the retry mechanism the worker advertises cannot do anything useful — it can only ever fail faster with a misleading "file not found" error masking the real root cause. Worse, by the time this happens the destructive `TRUNCATE` (run synchronously at upload time, in `migration.ts`) has *already* committed, so the user is left with an empty database, a stuck/failing job, and no way to recover via the existing queue-retry path (their only recourse is to re-upload the file from scratch).

**Fix:** Either (a) do not delete the file until the job reaches a terminal state that will not be retried (move the `unlink` out of `finally` and only call it when `finalStatus !== 'failed'` or when `errors.length === 0`, and additionally call it from `workerLoop`'s "max retries reached" branch), or (b) make `excel_migration` jobs non-retryable by archiving/deleting the message immediately regardless of outcome (since retrying a destructive migration from a half-wiped DB is not a safe operation anyway):

```ts
// in workerLoop, branch on payload.type:
if (payload.type === 'excel_migration') {
  // Excel migrations are not safely retryable (DB already wiped, file already
  // consumed) — always archive/delete the message after one attempt.
  await sql`SELECT pgmq.archive(${QUEUE_NAME}, ${msg.msg_id}::bigint)`;
} else {
  // existing retry logic for CSV imports
}
```

### CR-02: "Database transaction has been rolled back" message is false — partial data can persist after a destructive wipe

**File:** `frontend/src/pages/MigrationPage.tsx:251` (root cause: `src/workers/import-worker.ts:389-419`)
**Issue:** On failure, the UI tells the user:

```
An error occurred during spreadsheet processing. Database transaction has been rolled back. Error: {errorMessage}
```

This is not true. `processExcelMigrationJob` commits in independent transactions per batch of `EXCEL_INSERT_BATCH_SIZE = 100` rows, sheet by sheet:

```ts
for (let i = 0; i < transactions.length; i += EXCEL_INSERT_BATCH_SIZE) {
  const batch = transactions.slice(i, i + EXCEL_INSERT_BATCH_SIZE);
  await sql.begin(async (sql) => { /* insert each tx in `batch` */ });
  totalProcessed += batch.length;
  await sql`UPDATE import_jobs SET processed = ${totalProcessed} ...`;
}
```

If the worker crashes or a later batch/sheet fails after several batches have committed, those committed rows remain in the database — there is no overall rollback. Combined with the fact that the original ledger has *already* been irreversibly truncated (`migration.ts:39`), a failure can leave the user with neither their original data nor a complete migrated dataset, while the UI explicitly (and incorrectly) reassures them that a rollback occurred and the database is in a clean/consistent state.

**Fix:** Either make the wording accurate (e.g., "Some rows may have been imported before the error occurred; check the Transactions view and re-run the migration if needed — this operation cannot be safely retried automatically"), or change the implementation to match the promised guarantee by wrapping the entire ingestion in a single transaction (acceptable here because data volumes are bounded — a personal-finance ledger spanning a few years is at most a few thousand rows) so a failure genuinely rolls everything back:

```ts
await sql.begin(async (sql) => {
  for (const sheet of sheets) {
    // insert opening balance + all transactions for this sheet
  }
});
```

### CR-03: Empty/invalid workbook is reported as "completed" after destructively wiping the database

**File:** `src/workers/import-worker.ts:442-447`, surfaced via `frontend/src/pages/MigrationPage.tsx:219-222`
**Issue:** The final status is computed as:

```ts
const finalStatus = errors.length > 0 && totalProcessed === 0 ? 'failed' : 'completed';
```

If the uploaded workbook contains zero recognizable monthly sheets (e.g., the user accidentally uploads the wrong file, an empty template, or a workbook whose sheet names don't match any month pattern), `extractWorkbook` returns `[]`, `totalRows = 0`, `totalProcessed = 0`, and `errors = []`. Since `errors.length === 0`, the condition is false and `finalStatus = 'completed'`.

The frontend then renders the success screen:

> "Migration Completed — All historical sheets have been processed successfully. Existing data was wiped and replaced with historical ledger records from budget.xlsx."

…even though zero rows were imported and the previously-existing data has already been permanently destroyed by the `TRUNCATE` in `migration.ts:39`. This is the worst possible outcome for a "type MIGRATE to confirm" destructive operation: silent total data loss reported as success. (`tests/migration-api.test.ts:197-201` even encodes this ambiguity by accepting either `'failed'` or `'completed'` as a valid outcome for an invalid workbook.)

**Fix:** Treat "zero rows extracted from a non-empty upload" as a failure condition, and ideally validate the workbook structure *before* truncating (e.g., parse and sanity-check the file in the route handler — or in a pre-flight step inside the same transaction — before running `TRUNCATE`, so a bad upload never destroys good data):

```ts
const finalStatus = (totalProcessed === 0) ? 'failed' : 'completed';
if (totalProcessed === 0 && errors.length === 0) {
  errors.push('No recognizable monthly sheets or transactions were found in the uploaded workbook.');
}
```
And/or move workbook parsing ahead of the destructive `TRUNCATE` so an unparseable file never triggers data loss in the first place.

## Warnings

### WR-01: Uploaded file can be orphaned on disk if the enqueue transaction fails

**File:** `src/interface-adapters/api/migration.ts:30-65`
**Issue:** The file is written to disk (`Bun.write(filePath, arrayBuffer)`, line 35) *before* the `sql.begin` transaction that performs the destructive `TRUNCATE`, looks up the `IPKO` account, inserts the `import_jobs` row, and calls `pgmq.send`. If any step inside the transaction throws — e.g., `ipkoAccount` not found (line 47), or `sendResult` is unexpectedly `undefined` causing `Number(sendResult.msg_id)` (line 64) to throw `TypeError` — the transaction rolls back (good: the destructive truncate is undone), but the file already written at `filePath` is never cleaned up. Only the worker's `finally` block deletes the file, and the worker never runs without a queue message.

Repeated failed upload attempts (e.g., misconfigured environment where the seeded `IPKO` account is missing) will silently accumulate orphaned `.xlsx` files in `scratch/uploads`.

**Fix:** Wrap the write + transaction in a try/catch that deletes the just-written file on any failure path, or defer the disk write until after the transaction commits successfully:

```ts
try {
  const result = await sql.begin(async (sql) => { /* ... */ });
  return c.json({ data: { job_id: result.jobId }, ... }, 202);
} catch (err) {
  await Bun.file(filePath).delete().catch(() => {});
  throw err;
}
```

### WR-02: Frontend shows the wrong validation message for oversized files

**File:** `frontend/src/pages/MigrationPage.tsx:52-56`
**Issue:** `validateAndSetFile` uses the same error message for both the wrong-extension case and the file-too-large case:

```ts
if (!selectedFile.name.toLowerCase().endsWith('.xlsx')) {
  setFileError('Only .xlsx Excel workbooks are supported');
  ...
}
if (selectedFile.size >= MAX_FILE_SIZE_BYTES) {
  setFileError('Only .xlsx Excel workbooks are supported');  // <-- wrong message for a size problem
  ...
}
```

A user who drops a valid 25 MB `.xlsx` file is told "Only .xlsx Excel workbooks are supported", which is misleading — the file *is* an .xlsx workbook; the actual problem is its size.

**Fix:**
```ts
if (selectedFile.size >= MAX_FILE_SIZE_BYTES) {
  setFileError('File exceeds the 20MB size limit');
  setFile(null);
  return;
}
```

### WR-03: Frontend/backend file-size boundary check is inconsistent (`>=` vs `>`)

**File:** `frontend/src/pages/MigrationPage.tsx:52` vs `src/interface-adapters/api/migration.ts:23`
**Issue:** The frontend rejects files where `size >= MAX_FILE_SIZE_BYTES` (i.e., a file of exactly 20 MB is rejected), while the backend rejects only `size > MAX_FILE_SIZE_BYTES` (a file of exactly 20 MB is accepted). This is a minor off-by-one inconsistency between the two validation layers — a 20 MB-exactly file is rejected client-side but would have been accepted server-side, and the two limits should agree so the client-side check is a faithful preview of server behavior.

**Fix:** Use the same comparison operator in both places (prefer `>` to match the stated "Max size: 20MB" copy, which implies 20MB itself is allowed):
```ts
if (selectedFile.size > MAX_FILE_SIZE_BYTES) { ... }
```

### WR-04: No server-side content/type validation before destructive truncate — any file up to 20MB triggers the wipe

**File:** `src/interface-adapters/api/migration.ts:14-35`
**Issue:** The route only validates `file instanceof File` and `file.size <= MAX_FILE_SIZE_BYTES`. It does not check the file extension, MIME type, or magic bytes (e.g., the ZIP signature `PK\x03\x04` that `.xlsx` files start with). Any file — a text file, an image, a script renamed or not — up to 20 MB will be accepted, written to disk, and will trigger the destructive `TRUNCATE` inside the same transaction that enqueues the job, *before* the worker ever attempts to parse it. The worker will then fail with an `XLSX.read` parse error (caught, reported as `'failed'`), but the data has already been wiped (see CR-03 for the consequence when this combines with the "completed on zero rows" bug, and CR-01 for why retry can't help).

While a malicious *authenticated* user could already do far more damage with direct DB access, this endpoint specifically lacks the kind of pre-flight validation one would expect before an irreversible action — the modal asks the user to type "MIGRATE" to confirm, but the server will happily truncate the database for a `.txt` file too.

**Fix:** Validate the file's magic bytes / parseability with a lightweight pre-check (or perform the actual `XLSX.read`/`extractWorkbook` parse) *before* opening the destructive transaction, so an unparseable upload never reaches `TRUNCATE`:

```ts
const arrayBuffer = await file.arrayBuffer();
let workbook: XLSX.WorkBook;
try {
  workbook = XLSX.read(arrayBuffer, { type: 'array' });
} catch {
  return c.json({ data: null, error: { message: 'Uploaded file is not a valid Excel workbook' }, meta: null }, 400);
}
// ... only now proceed to write the file and open the destructive transaction
```

### WR-05: No request body size limit configured — `formData()` buffers the entire upload before the size check runs

**File:** `src/interface-adapters/api/migration.ts:16-25` (also affects the pre-existing `/import` route, but is especially relevant here given the destructive nature of this endpoint)
**Issue:** `c.req.formData()` (line 16) reads and buffers the entire multipart body into memory before any size validation occurs. The `MAX_FILE_SIZE_BYTES` check (line 23) only runs *after* the full payload — potentially much larger than 20 MB, since nothing caps the request body — has already been parsed and held in memory. There is no `bodyLimit` middleware configured anywhere in `index.ts`. An attacker (or a buggy client) could send an arbitrarily large multipart body, exhausting server memory, well before the application-level size check has a chance to reject it.

**Fix:** Add a body-size-limiting middleware (e.g., Hono's `bodyLimit`) ahead of the multipart parse, sized to the application's actual maximum (e.g., `MAX_FILE_SIZE_BYTES` plus a small multipart-overhead margin):

```ts
import { bodyLimit } from 'hono/body-limit';

migrationRoutes.post('/excel', requireAuth, bodyLimit({ maxSize: MAX_FILE_SIZE_BYTES + 1024 * 1024 }), async (c) => {
  ...
});
```

## Info

### IN-01: `getMigrationStatus` and `getImportStatus` are byte-for-byte duplicates

**File:** `frontend/src/api.ts:61-79`
**Issue:** Both functions have an identical implementation — they call `apiFetch(`/import/${jobId}`)`, check `res.ok`, and return `json.data`, differing only in the wording of their error messages:

```ts
export async function getMigrationStatus(jobId: string) {
  const res = await apiFetch(`/import/${jobId}`);
  ... throw new Error(errorJson.error?.message || `Failed to get migration status: ${res.statusText}`);
}

export async function getImportStatus(jobId: string) {
  const res = await apiFetch(`/import/${jobId}`);
  ... throw new Error(errorJson.error?.message || `Failed to get status: ${res.statusText}`);
}
```

This duplication means any future change to the polling/status logic (e.g., adding caching, retry-on-network-error, or switching to a dedicated `/api/migration/status/:id` endpoint) has to be made in two places, and the two call sites will silently drift.

**Fix:** Have `getMigrationStatus` delegate to `getImportStatus` (or vice versa), or define a single shared `getJobStatus` helper:
```ts
export async function getMigrationStatus(jobId: string) {
  return getImportStatus(jobId);
}
```

### IN-02: Internal error messages are returned verbatim to the authenticated client on this highly-sensitive endpoint

**File:** `src/interface-adapters/api/migration.ts:68-71`
**Issue:** The catch-all handler returns `err.message` directly in the JSON response body:
```ts
} catch (err) {
  const message = err instanceof Error ? err.message : 'Internal server error';
  return c.json({ data: null, error: { message }, meta: null }, 500);
}
```
For a route that performs a destructive `TRUNCATE` and touches the filesystem, raw exception messages (e.g., Postgres errors mentioning table/constraint names, filesystem paths, `TypeError: Cannot read properties of undefined (reading 'msg_id')`) could leak internal implementation details. This mirrors the pre-existing pattern in `import.ts`, but is worth tightening here given the heightened sensitivity of this endpoint.

**Fix:** Log the full error server-side and return a generic message to the client:
```ts
} catch (err) {
  console.error('[migration] Excel migration upload failed:', err);
  return c.json({ data: null, error: { message: 'Failed to start migration' }, meta: null }, 500);
}
```

### IN-03: `total_rows: 0` and zero-progress states are visually indistinguishable from "not yet started" in the progress UI

**File:** `frontend/src/pages/MigrationPage.tsx:169-174`
**Issue:**
```ts
const total = job?.total_rows || 0;
const processed = job?.processed || 0;
const percent = total > 0 ? (processed / total) * 100 : 0;
const details = total > 0 ? `Processing row ${processed} of ${total}` : 'Preparing workbook...';
```
A workbook that genuinely contains zero transactions (see CR-03) will show "Preparing workbook..." indefinitely (or until the job flips to `'completed'`/`'failed'`), which is indistinguishable from the normal startup phase where `total_rows` hasn't been computed yet. This is a minor UX nit that becomes more visible once CR-03 is fixed and such workbooks correctly transition to `'failed'`.

**Fix:** No code change strictly required once CR-03 is fixed (the state will resolve to `'failed'` quickly), but consider distinguishing "computing total rows" from "zero rows found" if the job stays in `processing` for an extended period.
