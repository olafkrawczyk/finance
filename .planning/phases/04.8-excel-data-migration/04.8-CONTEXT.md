# Phase 04.8: Excel Data Migration - Context

**Gathered:** 2026-06-07
**Status:** Ready for planning
**Source:** PRD Express Path (analysis_results.md)

<domain>
## Phase Boundary

This phase delivers a dedicated data migration feature in the application to ingest historical budget data from an Excel spreadsheet (`budget.xlsx`).

### Deliverables
1. **Backend Excel Ingestion Service:** An endpoint (`POST /api/migration/excel`) that accepts an uploaded Excel file, performs a destructive wipe of existing transaction and opening balance data, parses all monthly sheets from the spreadsheet, maps legacy categories and routes transactions to correct accounts (ING vs. PKO) based on predefined rules, and writes all records to the database.
2. **Migration UI Page:** A dedicated route `/migration` (hidden from the navigation header) providing a drag-and-drop file uploader, a prominent confirmation warning dialogue alerting users to the destructive nature of the operation, and real-time status/progress tracking of the migration job.

</domain>

<decisions>
## Implementation Decisions

### Ingestion Trigger & Cleanup
- **Destructive Reset:** Before starting the ingestion, the backend must truncate `transactions`, `monthly_opening_balances`, `insights`, and `import_jobs` tables (CASCADE) to ensure a clean initialization point.
- **Async Queue Flow:** Excel processing must run as an async job using the queue system. Uploading the file enqueues a job to the background worker to avoid blocking the HTTP request handler.
- **UX Destructive Alert:** The frontend must show a clear, destructive-action warning dialog before allowing the user to initiate the migration.

### Spreadsheet Parsing & Layout Adaptation
- **Sheet Processing:** The parser must process all monthly sheets in order, starting from the oldest (`lipiec` 2020) to the newest (`pazdziernik2025`). Skip helper sheets like `kategorie` and `Zbiorczy`.
- **Ignore Redundant Data:** Columns R–U in the monthly sheets contain duplicates of the previous month's transactions. The parser must completely ignore columns R–U.
- **Legacy Sheets (Jul 2020 – Jan 2021):** 
  - Expense category is in column A, amount is in column B. 
  - Date and description are missing; default the date to the 1st of the month (e.g. `2020-07-01` for `lipiec`), and description to `NULL`.
  - Incomes: Columns D–E contain income name and amount (July to October 2020); columns E–F contain income name and amount (November 2020 to January 2021).
  - Opening Balance: Column G, Row 3 contains the opening balance (July to October 2020); Column H, Row 3 contains it for November 2020 to January 2021.
- **Modern Sheets (Feb 2021 – Oct 2025):**
  - Expenses: Columns A–D contain category, amount, description, and date.
  - Incomes: Columns E–F contain income name and amount.
  - Opening Balance: Column H, Row 3 contains the opening balance.

### Account Routing Rules
Transactions must be automatically routed to their corresponding database accounts:
- **ING Business (`Konto Direct dla Firmy`)** if:
  - Category is `VAT`, `ZUS`, `PIT`/`PIT36`, or `paliwo`.
  - Category is `auto` AND the transaction amount is strictly greater than 2000 PLN.
  - Description contains `PPE`, `ORANGE`, or `PLAY` (case-insensitive).
- **PKO Personal (`IPKO`)** for:
  - All other transactions (including incomes unless they meet the business criteria above).

### Category Mappings
Legacy categories must be translated on-the-fly:
- `dentysta` ➔ `lekarz`
- `mpk` ➔ `przejazdy`
- `kawka` ➔ `kawa`
- Any other unrecognized category ➔ Fallback to `fun`

### the agent's Discretion
- Selection of the Excel parsing library (`xlsx` npm package).
- Design of the background job message format and job polling database updates.
- Styling of the migration page (using CSS / custom components matching the FinanceFlow branding).

</decisions>

<specifics>
## Specific Ideas
- The system should store the uploaded `.xlsx` file temporarily on disk or parse it in memory, or pass it via base64 in the PGMQ message. Given the file size is ~1MB, saving to a temporary directory in the workspace (e.g. `scratch/uploads/`) is a clean approach.
- The UI should display progress metrics (e.g., "Sheet 5/60 processed") to keep the user informed during the long-running migration.

</specifics>

<deferred>
## Deferred Ideas
- Dynamic mapping of arbitrary unrecognized categories from the UI (out of scope, fallback is hardcoded to `fun` for now).
- Rolling back a migration to restore old data (destructive wipe is final; database backup/restore is deferred to dev operations).

</deferred>
