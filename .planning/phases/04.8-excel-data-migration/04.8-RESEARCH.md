# Phase 04.8: Excel Data Migration - Research

This document outlines the domain analysis, implementation approach, technical details, dependencies, and validation architecture for Phase 4.8: Excel Data Migration.

---

## 1. Domain Analysis

### Phase Boundary
The goal of this phase is to import historical financial ledger data from an Excel spreadsheet (`budget.xlsx`) spanning years of monthly records. This ingestion is a one-time setup mechanism that replaces the active ledger state with historical data.

### Core Entities & Database Targets
1. **`transactions`**: Historical income, expense, and transfer records.
2. **`monthly_opening_balances`**: Net worth/opening balance at the start of each month.
3. **`import_jobs`**: Job status tracking for progress polling in the UI.

### Data Layout Shifts & Structural Formats
The spreadsheet spans July 2020 to October 2025 (64 months/sheets). Layout and column structures shift over time:
- **Legacy Sheets (July 2020 – January 2021)**:
  - **Expenses**: Category in Column A, Amount in Column B. No date or description exists; dates default to the 1st of the month, and descriptions default to `NULL`.
  - **Incomes**: Names and amounts are in Columns D–E (July to October 2020) or Columns E–F (November 2020 to January 2021).
  - **Opening Balance**: Contained in Row 3, Column G (July to October 2020) or Row 3, Column H (November 2020 to January 2021).
- **Modern Sheets (February 2021 – October 2025)**:
  - **Expenses**: Columns A–D contain category, amount, description, and date.
  - **Incomes**: Name in Column E, Amount in Column F.
  - **Opening Balance**: Contained in Row 3, Column H.
- **Redundancy Filter**: Columns R–U in all monthly sheets contain redundant copies of the previous month's data and must be completely ignored.

### Business Rules & Mappings
- **Category Translations**:
  - Legacy categories must translate on-the-fly: `dentysta` ➔ `lekarz`, `mpk` ➔ `przejazdy`, `kawka` ➔ `kawa`.
  - Any category name not matching the seeded 26 application categories defaults to `fun`.
- **Account Routing Rules**:
  - Route to **ING Business** (`Konto Direct dla Firmy`) if:
    - Mapped category is `VAT`, `ZUS`, `PIT36` (or contains `PIT`), or `paliwo`.
    - Mapped category is `auto` AND the transaction amount is strictly greater than 2000 PLN.
    - Description contains `PPE`, `ORANGE`, or `PLAY` (case-insensitive).
  - Route to **PKO Personal** (`IPKO`) for all other transactions (including incomes unless they meet the ING criteria above).
- **Incomes**:
  - Imported as `type = 'income'`, mapping name as description and amount as value, with date defaulting to the 1st of the month. Category is set to `NULL`.

## 2. Implementation Approach

### Backend Pipeline (`POST /api/migration/excel`)
1. **Request Handler**:
   - Requires user authentication via Hono session middleware.
   - Accepts multipart form data with the uploaded file.
   - Generates a UUID for the migration job and writes the binary buffer to `scratch/uploads/${jobId}.xlsx`.
2. **Destructive Reset**:
   - Performs an atomic `TRUNCATE transactions, monthly_opening_balances, insights, import_jobs CASCADE` to wipe out any active ledger data. User sessions and categories are preserved.
3. **Queue Enqueuing**:
   - Creates a job record in `import_jobs` with status `pending`.
   - Enqueues a message to PGMQ (`import_queue`):
     ```json
     {
       "job_id": "job-uuid",
       "type": "excel_migration",
       "file_path": "/home/olafk/finance/scratch/uploads/job-uuid.xlsx"
     }
     ```
   - Returns a `202 Accepted` status with `{ job_id }` immediately.

### Worker Processing (`src/workers/import-worker.ts`)
1. **Payload Dispatch**:
   - Extend the worker's processing function to detect `type === 'excel_migration'`.
   - Set status to `processing`.
2. **Sheet Ingestion**:
   - Load the workbook from the specified file path using the `xlsx` package.
   - Extract and normalize sheet names (removing diacritics and converting to lowercase). Filter out non-month sheets (`zbiorczy`, `kategorie`).
   - Order the sheets chronologically (from July 2020 to October 2025).
3. **Database Insertion**:
   - For each sheet:
     - Parse the opening balance cell and write to `monthly_opening_balances` (`year`, `month`, `opening_balance`).
     - Parse expenses and incomes. Translate categories and resolve account routing.
     - Build unique `import_hash` values for each record.
     - Perform batch inserts into the `transactions` table.
     - Update progress status in `import_jobs` (e.g., sheet processing count or row count).
   - Upon completion, delete the temporary file and set the job status to `completed` or `failed` (capturing any errors).

### Frontend Migration Page (`/migration`)
- A dedicated page at path `/migration`, excluded from the header navigation menu.
- Displays a prominent destructive warning alerting the user that initiating migration permanently deletes all existing transaction records.
- Provides a drag-and-drop file uploader accepting `.xlsx` files.
- Once submitted, triggers `POST /api/migration/excel` and transitions to a real-time progress screen, polling `/api/migration/excel/:job_id` or reusing `/import/:job_id` to show progress percentages, processed sheets, and errors.

## 3. Technical Details

### Excel Date & Number Parsing
- Standard JS timezone shifts can distort Excel serial dates. To be timezone-safe, parse Excel serial dates using timezone-independent arithmetic:
  `XLSX.SSF.parse_date_code(dateValue)` returns `{y, m, d}` directly.
- Normalize sheet names with diacritics mapping:
  ```typescript
  function normalizeText(str: string): string {
    return str
      .toLowerCase()
      .normalize('NFD')
      .replace(/[\u0300-\u036f]/g, '')
      .replace(/ł/g, 'l')
      .replace(/Ł/g, 'l');
  }
  ```
- Map normalized month names to month numbers (1-12). If the sheet name lacks a year suffix, map legacy names (`lipiec` to `styczen`) to `2020` or `2021` according to the legacy boundary.

### Deduplication & Integrity
- Since the Excel migration is a complete historical dump, we must import every single row. To prevent database unique constraint collisions on `import_hash` while maintaining deduplication constraints, generate a unique hash containing the Excel sheet and row index:
  `import_hash = SHA-256 of ${sheetName}|${rowNumber}|${date}|${amount}|${description}`.

### Performance & Safety
- **Transaction batching**: Use transaction blocks (e.g. per-sheet or in batches of 100-200 rows) to perform bulk inserts efficiently.
- **Cleanup**: Delete temporary `.xlsx` files from `scratch/uploads/` when the job completes or fails to prevent disk bloat.

## 5. Validation Architecture

### Unit Testing (Excel Parsing & Ingestion Logic)
- **File: `tests/migration-parser.test.ts`**
- Use `xlsx` utility functions to build mock workbooks in memory (converting AOA - arrays of arrays - to sheets) containing sample rows for:
  - July 2020 (Legacy layout, July–October 2020 income columns).
  - November 2020 (Legacy layout, November–January 2021 income columns).
  - February 2021 (Modern layout).
- Verify:
  - Sheet name chronological sorting.
  - Legacy category mappings (`dentysta` ➔ `lekarz`, unrecognized ➔ `fun`).
  - Account routing decisions (e.g., category triggers, amount thresholds like `auto > 2000` -> ING, description overrides like `ORANGE` -> ING).
  - Correct extraction of opening balances.

### Integration Testing (API & Worker Pipeline)
- **File: `tests/migration-api.test.ts`**
- Test that:
  - `POST /api/migration/excel` truncates the database tables, creates an `import_jobs` record, and returns a job ID.
  - The queue worker picks up the job, processes it, creates the corresponding transactions/opening balances in the database, and updates status to `completed`.
  - Corrupt or invalid workbook files transition the job status to `failed` and populate error lists.
