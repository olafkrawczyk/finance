# Phase 04.8 Plan 01: Backend Excel Ingestion Service Summary

Implements the complete backend pipeline for ingesting the historical `budget.xlsx` workbook: a Hono `/api/migration/excel` route that authenticates, uploads, destructively wipes the ledger in a transaction, and enqueues a PGMQ job; a chronological Excel parsing/mapping engine (`excel-parser.ts`) handling legacy and modern sheet layouts, Polish text normalization, category translation, and ING/PKO account routing; and a worker integration that loads the workbook, ingests opening balances and transactions in batches, and always cleans up the temp file.

### Task 2 ŌĆö Excel Parser and Mapping Rule Engine
Created `src/core/import/excel-parser.ts`:
- `normalizeText`: lowercases, NFD-decomposes, strips combining diacritics (`╠Ć-═»`), maps Polish ┼é/┼ü ŌåÆ l
- `resolveSheetMeta` / `sortSheetsChronologically`: extracts `{year, month}` from sheet names, mapping legacy no-year sheets (lipiecŌĆōgrudzien ŌåÆ 2020, styczen ŌåÆ 2021), skipping `zbiorczy`/`kategorie` helper sheets, sorting `(year, month)`
- `parseExcelDate`: timezone-safe conversion via `XLSX.SSF.parse_date_code` for numeric serials, regex parsing for `YYYY-MM-DD`/`DD.MM.YYYY` strings, defaulting to the 1st of the sheet month
- `resolveOpeningBalanceColumnIndex`: Column G (index 6) for JulŌĆōOct 2020, Column H (index 7) from Nov 2020 onward, reading Row 3 (index 2)
- `resolveCategory`: applies legacy translations (`dentysta`ŌåÆ`lekarz`, `mpk`ŌåÆ`przejazdy`, `kawka`ŌåÆ`kawa`) then matches against seeded DB categories case/diacritic-insensitively, falling back to `fun`
- `routeAccount`: routes to ING Business (`Konto Direct dla Firmy`) for VAT/ZUS/PIT/PIT36/paliwo categories, `auto` + amount > 2000 PLN, or descriptions containing PPE/ORANGE/PLAY (case-insensitive); all else to PKO Personal (`IPKO`)
- `parseSheetTransactions` / `extractWorkbook`: full modern (Cols AŌĆōD expenses, EŌĆōF incomes) and legacy (Cols AŌĆōB expenses, DŌĆōE or EŌĆōF incomes by date range) parsing, RŌĆōU column filtering, header-row skipping
- `buildMigrationImportHash`: SHA-256 of `sheetName|rowNumber|date|amount|description`

Replaced the parser test stubs with 20 real assertions ŌĆö all passing.

### Task 3 ŌĆö Destructive Reset & Ingestion REST API Route
Created `src/interface-adapters/api/migration.ts`:
- `POST /excel`: requires auth (401 if anonymous), accepts multipart `file`/`excel` field with 20MB limit, generates a server-side `crypto.randomUUID()` job ID (never trusts client input ŌĆö T-MIG-04), writes the upload to `scratch/uploads/<uuid>.xlsx` via `Bun.write`
- Wraps `TRUNCATE transactions, monthly_opening_balances, insights, import_jobs CASCADE`, the PKO Personal account lookup, the pending `import_jobs` insert, and `pgmq.send('import_queue', {job_id, type: 'excel_migration', file_path})` in a single `sql.begin` transaction
- Returns `202 Accepted` with `{ data: { job_id }, error: null, meta: null }`

Mounted `migrationRoutes` at `/api/migration` in `index.ts`, and added the `/api/migration` proxy entry to `vite.config.ts`. Added `scratch/` to `.gitignore`.

### Task 4 ŌĆö Worker Job Processing Integration & Cleanup
Extended `src/workers/import-worker.ts`:
- `processJob` now dispatches on `payload.type === 'excel_migration'` to the new `processExcelMigrationJob`, otherwise delegates to the original LLM/CSV pipeline (renamed internally to `processCsvImportJob` ŌĆö fully preserved, zero behavior change, existing `import-worker.test.ts` still passes 2/2)
- `processExcelMigrationJob`: sets status `processing`, loads DB categories, resolves ING Business / PKO Personal account IDs (throwing if either is missing), loads the workbook from `payload.file_path` via `xlsx`, calls `extractWorkbook` to get chronologically-sorted sheets, computes and stores `total_rows`, then iterates sheets inserting opening balances (`ON CONFLICT (year, month) DO NOTHING`) and batch-inserting transactions (100/batch, `ON CONFLICT (import_hash) DO NOTHING`) with dynamic `processed`/`updated_at` progress updates
- Sets status `completed` on success, `failed` with appended error messages on per-sheet or fatal errors
- **Always** deletes the temp upload file in a `finally` block, tolerating `ENOENT` (T-MIG-05)

Replaced the API/worker test stubs with 4 real integration tests covering: 401 for anonymous requests, full uploadŌåÆ202ŌåÆjob_idŌåÆPGMQ-enqueueŌåÆdestructive-wipe flow, end-to-end worker processing (opening balance insertion, VATŌåÆING routing, funŌåÆPKO routing, income with NULL category, temp-file cleanup), and failed-status handling for structurally invalid workbooks. All pass.

## Threat Flags

No new surface beyond what the plan's `<threat_model>` already registers (T-MIG-01 through T-MIG-05). All five threats were mitigated as specified:

- T-MIG-03 (SQL Injection): all queries use the `sql` tagged-template parameterized interface (no string concatenation)
- T-MIG-04 (Path Traversal): `crypto.randomUUID()` generates server-side job IDs; files saved strictly to `scratch/uploads/<uuid>.xlsx`
- T-MIG-05 (Temp file bloat): `finally` block in `processExcelMigrationJob` deletes the upload on both success and failure paths
