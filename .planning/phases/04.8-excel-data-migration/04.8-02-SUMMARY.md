# Phase 04.8 Plan 02: Frontend Ingestion UI Summary

Builds the hidden `/migration` route end-to-end: a drag-and-drop `.xlsx` uploader with full validation states, a destructive-action confirmation modal requiring an exact "MIGRATE" text challenge, and a 2000ms-polling progress/success/failure state machine — all styled in pure Tailwind CSS v4 per the locked 60/30/10 dark-slate design contract with copywriting matching the UI-SPEC verbatim.

## What Was Built

### Task 1 — API client endpoints and hidden route registration
Added two functions to `frontend/src/api.ts`:
- `startExcelMigration(file)`: builds a `FormData` with the file under the `file` key, POSTs to `/api/migration/excel` with `credentials: include` (via the shared `apiFetch` wrapper), throws on non-OK responses, and returns `{ job_id }`.
- `getMigrationStatus(jobId)`: GETs `/import/:jobId` (the existing `import_jobs` status endpoint shared with CSV imports) and returns the `ImportJob`-shaped object.

Registered `/migration` in `App.tsx`: imported `MigrationPage` and added a `currentPath === '/migration'` branch inside `renderContent()`, placed within the authenticated layout (after the auth-guard checks) and intentionally **not** added to the `<nav>` header links — keeping it hidden per REQ-7.1. Passes `onMigrationComplete={() => navigateTo('/dashboard')}` so the success screen's "Go to Dashboard" button uses the app's internal navigation rather than a hard page reload.

### Task 2 — DestructiveConfirmModal and ProgressBar components
Created `frontend/src/components/DestructiveConfirmModal.tsx`:
- Backdrop (`fixed inset-0 bg-slate-950/80 backdrop-blur-sm z-50 flex items-center justify-center p-4`) and modal card (`bg-slate-900 border border-slate-800 rounded-xl p-8 max-w-md w-full shadow-2xl space-y-6`) exactly per spec.
- Red warning panel (`bg-red-950/50 border border-red-800 text-slate-100 rounded-lg p-4 flex gap-3 items-start`) with an inline alert-triangle SVG icon.
- All five copywriting strings (Warning Header, Warning Message, Challenge Label, Submit Button, Cancel Button) reproduced character-for-character from the UI-SPEC contract, including the `⚠️` emoji.
- Controlled text input (`border border-slate-800 bg-slate-950 text-slate-100 px-3 py-2 rounded focus:outline-none focus:border-blue-500 w-full`) gating the destructive submit button: `disabled` + `opacity-50 cursor-not-allowed` until the value equals `MIGRATE` exactly (case-sensitive, no trim).
- Cancel resets the challenge input state and calls `onClose()`.

Created `frontend/src/components/ProgressBar.tsx`: horizontal track (`w-full bg-slate-950 rounded-full h-3 overflow-hidden border border-slate-800`) with a gradient fill (`bg-gradient-to-r from-blue-600 to-indigo-600 h-full transition-all duration-300`), width driven by a clamped `percent` prop (0–100), displaying the rounded percentage and an optional `details` subtext line.

### Task 3 — MigrationPage and polling state machine
Created `frontend/src/pages/MigrationPage.tsx` implementing all four UI states as a single component with a `pageState` discriminated union (`idle | processing | success | failed`):

1. **Idle/Dropzone**: Page shell `bg-slate-950 text-slate-100 min-h-screen p-8 flex flex-col items-center justify-center`; dropzone `border-2 border-dashed border-slate-800 bg-slate-900/50 p-12 rounded-xl text-center cursor-pointer max-w-xl w-full flex flex-col items-center gap-4 transition-all`. Drag-over swaps to `border-blue-500 bg-blue-950/20`; invalid-file state swaps border to `border-red-800` and shows the exact rejection copy `Only .xlsx Excel workbooks are supported`. Valid selection shows filename, human-readable size (KB/MB), and a trash-icon "Remove file" button. "Start Ingestion" is disabled until a valid file is selected.
2. **Modal trigger**: clicking "Start Ingestion" opens `DestructiveConfirmModal`.
3. **Upload & polling**: on confirm, the modal closes, `startExcelMigration(file)` is called, and on success the page transitions to `processing` and begins polling `getMigrationStatus(jobId)` every 2000ms via `setInterval` (with an immediate first fetch and ref-based cleanup on unmount, mirroring `ImportStatus.tsx`'s established pattern).
4. **Processing screen**: animated spinner SVG, header `Ingesting Excel Data...`, subtext `Please keep this tab open and do not navigate away. This may take up to a minute.`, and a `<ProgressBar>` computing `(processed / total_rows) * 100` with a live row-count details label.
5. **Success screen** (`status === 'completed'`): stops polling, renders the success card (`bg-slate-900 border border-slate-800 p-8 rounded-xl max-w-md text-center space-y-4`), green checkmark SVG, header `Migration Completed`, exact subtext, and `Go to Dashboard` CTA wired to `onMigrationComplete` (navigates to `/dashboard` via the app's internal router).
6. **Failed screen** (`status === 'failed'` or request error): stops polling, renders the failure card with a red alert-triangle SVG, header `Migration Failed`, subtext interpolating the last error message (`Error: {error_message}`), and a `Try Again` CTA that fully resets state back to the idle dropzone (clears file, errors, job, and the file-input DOM value).

All copywriting strings (page subtext, dropzone labels, processing/success/failure headers and messages, button labels) were transcribed verbatim from `04.8-UI-SPEC.md`'s Copywriting Contract table.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Corrected migration upload endpoint path**
- **Found during:** Task 1 implementation
- **Issue:** A literal reading of the plan's action text could imply POSTing to `/migration/excel` relative to the existing `apiFetch` base, but `index.ts` mounts `migrationRoutes` at `/api/migration` (per `04.8-01-SUMMARY.md`) and `vite.config.ts` only proxies the `/api/migration` prefix (not a bare `/migration`)
- **Fix:** Used `/api/migration/excel` as the fetch target, matching the actual mounted route and the registered dev-server proxy entry, and matching the plan objective's own literal text ("Perform a `fetch` `POST` request to `/api/migration/excel`")
- **Files modified:** `frontend/src/api.ts`
- **Commit:** `194b0e1`

**2. [Rule 1 - Architecture adaptation] Registered the route via the existing custom minimal router, not React Router `<Route>` JSX**
- **Found during:** Task 1 implementation, after reading `App.tsx`
- **Issue:** The plan's action text says to "Add a `<Route path="/migration" element={<MigrationPage />} />`", but this codebase does not use `react-router-dom` — `App.tsx` implements its own minimal router via `currentPath` state, `window.history.pushState`, and a chain of `if (currentPath === ...)` branches inside `renderContent()`
- **Fix:** Added `if (currentPath === '/migration') { return <MigrationPage onMigrationComplete={...} />; }` inside `renderContent()`, in the same style as every other route in the file. This is functionally equivalent (auth-gated, hidden from `<nav>`) and consistent with the existing codebase convention; introducing `react-router-dom` would have been an unrequested architectural change
- **Files modified:** `frontend/src/App.tsx`
- **Commit:** `194b0e1`

### Notes (not deviations)
- The plan's Processing-state spec offered two alternative live-detail formats: a sheet-name-based label (`Processing Sheet: {sheetName} ({processedSheets} of {totalSheets})`) or a row-count-based label ("or processed row count label details" / "or sheet processed ratio if sheet count is returned"). Inspection of `schema.sql` and `import-worker.ts` confirmed the `import_jobs` table and the `getMigrationStatus`/`/import/:jobId` endpoint expose only `status`, `total_rows`, `processed`, and `errors` — no per-sheet tracking columns exist in the backend. The row-count-based fallback (`Processing row {processed} of {total}`) was therefore used, which the plan explicitly anticipated and permitted.

## Known Stubs

None — all states (idle, processing, success, failed) are wired to real API calls (`startExcelMigration`, `getMigrationStatus`) and real component state. No hardcoded/mock data flows into the rendered UI.

## Threat Flags

No new surface beyond what the plan's `<threat_model>` already registers (T-UI-01 through T-UI-03). All three threats were mitigated as specified:
- T-UI-01 (Challenge Bypass): submit button `disabled` attribute is gated on `challengeInput === 'MIGRATE'` (exact string equality, no normalization/trim that could be exploited); server-side auth is enforced independently by the existing `/api/migration/excel` route (verified in 04.8-01)
- T-UI-02 (File Type Bypass): `validateAndSetFile` rejects any filename not ending in `.xlsx` (case-insensitive) and any file `>= 20MB`, both on drop and on file-input change; the hidden `<input type="file" accept=".xlsx">` also constrains the OS file picker
- T-UI-03 (CSRF / Session Hijacking): both new API functions route through the shared `apiFetch` wrapper, which always sets `credentials: 'include'` and redirects to `/login` on a 401
