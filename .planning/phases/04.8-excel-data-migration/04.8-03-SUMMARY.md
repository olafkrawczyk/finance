# Phase 04.8 Plan 03: Gap Closure — Worker Reliability Fixes Summary

**Input validation guard, retry-safe temp file cleanup, atomic outer transaction, and empty-workbook detection closing all four UAT-identified gaps in the Excel migration worker**

## Accomplishments

- **GAP-01 — Input validation guard:** Added `typeof csv_content !== 'string' || csv_content.length === 0` guard in `processCsvImportJob` before `preprocessIngCsv`/`preprocessIpkoCsv` calls. Prevents the cryptic `content.split is not a function` TypeError when an old worker process (pre-type-routing) receives an `excel_migration` payload lacking `csv_content`.
- **GAP-02 — Retry-safe temp file cleanup:** Removed the unconditional `finally { unlink(file_path) }` block that guaranteed every PGMQ retry would fail with ENOENT. Temp file is now deleted only on terminal states: (a) on `'completed'` status in `processExcelMigrationJob`, (b) on max-retries-exhausted in the `workerLoop` catch block for `excel_migration` payloads.
- **GAP-03 — Atomic outer transaction:** Wrapped all sheet data operations (opening balances + batch transaction inserts) in a single `sql.begin(async (tx) => ...)` outer transaction. Removed per-batch `sql.begin` wrappers. Any failure inside the outer transaction rolls back ALL data — matching the UI's "Database transaction has been rolled back" guarantee. Progress/status updates (`total_rows`, `processed`, `status`) are performed outside the transaction.
- **GAP-04 — Empty workbook detection:** Added `totalRows === 0` guard that sets status to `'failed'` with error message `'No parseable data found in workbook'` before any transaction or data wipe would proceed. This prevents the silent data-loss scenario where an empty/invalid workbook was reported as `'completed'`.

## Decisions Made

- **Temp file cleanup in terminal states only:** The unconditional `finally` block was replaced with conditional cleanup that only fires on `'completed'` or max-retries-exhausted. This ensures PGMQ retries can re-read the temp file on non-terminal failures.
- **Outer transaction for atomicity:** All sheet data operations are wrapped in a single `sql.begin`. Per-batch `sql.begin` wrappers were removed as redundant. Progress tracking SQL updates intentionally kept outside the transaction.
- **Sheet error propagation:** Per-sheet try/catch blocks re-throw with a wrapped message (`Sheet ${sheetName}: ${errMsg}`) instead of logging and continuing. This makes any sheet failure trigger a full outer transaction rollback.
- **Early-return zero-row guard:** Empty workbook detection happens before the outer transaction, using a simple early-return with DB status update. No temp file deletion for non-terminal failure (consistent with GAP-02).

## Issues Encountered

- **Variable naming conflict in outer transaction:** The inner loop variable `tx` (short for transaction object) conflicted with the outer `sql.begin` callback parameter `tx` (SQL tagged template function). Renamed loop variable to `txn` to resolve the `Object is not a function` SQL error.
- **Interleaving of GAP-02 and GAP-04 test assertions:** The test change for GAP-02 (file-preserved-on-failure) requires GAP-04's empty-workbook guard to produce `status = 'failed'` before the file-preservation assertion is meaningful. Without GAP-04, an invalid workbook produces `status = 'completed'`, which triggers the success-path file cleanup. Both tasks were implemented together to resolve this ordering dependency.
