# Phase 4.9: Transaction List Enhancements & Reversal - Context

**Gathered:** 2026-06-07T10:54:20Z
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase delivers user interface enhancements for the transaction list and the monthly aggregates view:
1. In-memory, client-side searching, sorting, and filtering for the monthly transaction list (`MonthlyPage.tsx`).
2. Reversing the month order on the monthly summary ("zbiorczo") view to show the newest month first, while keeping the chronological balance calculations mathematically correct.

</domain>

<decisions>
## Implementation Decisions

### Filter UI Layout (MonthlyPage)
- **D-01:** Implement a grid layout directly above the transaction table that is always visible (no toggle or collapsible panel). This includes search input, type dropdown, category dropdown, and sort select.

### Default Transaction Sorting
- **D-02:** When entering the monthly view, the transaction list should sort by Date: Newest first (descending) by default.

### Search Matching Criteria
- **D-03:** The text search input must perform case-insensitive matching against both the `description` and `category_name` fields.

### Summary Reversal Level
- **D-04:** Implement API-level reversal in the backend use-case `getMonthlySummary()`. The running balance calculation will continue to run chronologically (ascending), but the returned array will be reversed before sending the JSON response.

### the agent's Discretion
- Visual styling details of the filter controls (e.g. padding, border radii, focus colors, icons) should reuse Tailwind styles from existing panels for consistency.

</decisions>
