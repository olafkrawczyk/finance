## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (page titles) | 24px (text-2xl) | 600 (font-semibold) | 1.25 (tight) |
| Heading (section titles) | 18px (text-lg) | 600 (font-semibold) | 1.25 (tight) |
| Body (content, descriptions) | 14px (text-sm) | 400 (font-normal) | 1.5 (relaxed) |
| Label (form labels, table headers) | 12px (text-xs) | 600 (font-semibold) | 1.25 (tight) |
| UI (buttons, nav, table cells) | 14px (text-sm) | 600 (font-semibold) or 500 (font-medium) | 1.4 |
| Metric (numbers, amounts) | 14px (text-sm) or 16px (text-base) | 700 (font-bold) | 1.25 (tight) |

**Line height defaults:**
- Body / paragraph text: `leading-relaxed` (1.5)
- Headings: `leading-tight` (1.25)
- Labels / metadata: `leading-tight` (1.25)
- Monospace numbers: `font-mono` (tabular figures, consistent width)

**Usage constraint:** Maximum 2 weights per element — `font-semibold` (600) for headings/buttons, `font-bold` (700) for emphasis on CTAs and numerical values. `font-medium` (500) appears in existing code for nav items and table cells — keep as-is. Do **not** introduce `font-light` (300) or `font-extrabold` (800).

**Pre-populated from:** Codebase scan — `text-sm` (14px) is the dominant body size across all pages. Page titles consistently use `text-2xl font-semibold`. Section headings use `text-lg font-semibold`. Table headers and labels use `text-xs font-semibold uppercase tracking-wider`.

Located at: `frontend/src/components/Skeleton.tsx`

| Prop | Type | Default | Description |
|------|------|---------|-------------|

- Border radius: `rounded-lg` (8px) — matches existing component card corners
- Height when no className: `h-4` (16px — standard text line height)

**Structural skeleton layouts** (D-18) — not separate components, but inline skeleton markup per page replicating the page's card/table/grid layout:

| Page | Skeleton Pattern |
|------|-----------------|
| Dashboard | 2 summary tile cards + 4 chart card placeholders (grid: 2 cols on lg) |
| Monthly | Filter bar placeholder + table header + 5-6 table row skeletons + sidebar card |
| Summary | Table header + 5-6 table row skeletons |

## Interaction Contract

### Cache Manager (D-10)
- **No visual footprint** — the `CacheManager` component renders `null`
- Triggers `queryClient.clear()` when `authClient.useSession()` transitions: `null → session` (login) or `session → null` (logout)
- After clear, all active `useQuery` hooks re-fetch automatically, triggering skeleton display for `isPending` queries

### Query Key Scoping (D-05, D-06)
- **Invisible to user** — no visual or interaction change
- All query keys follow `['user', userId, resource, ...filters]` pattern
- Cache per-user is automatically separated; no user-facing indication of cache scoping

### Auth Change UX
- Before: User sees existing page data (existing behavior)
- During auth transition cache clear + re-fetch: Loading skeletons displayed in place of each page's data regions
- After re-fetch completes: Data renders normally (no visual change from current behavior)
- **Critical timing constraint:** Skeleton display after cache clear must be instantaneous — no blank flash between `clear()` and skeleton render. Achieved by React Query `isPending` being immediately `true` on mount after cache clear.

## Interaction Edge Cases

### 1. Rapid auth changes (login → logout → login)
- Cache Manager fires `clear()` on each transition
- `clear()` is synchronous and idempotent — safe to call in rapid succession
- Each transition triggers skeleton → re-fetch cycle
- **Acceptable UX:** Brief skeleton flash (sub-500ms for cached data, API call time for fresh data)

### 2. Multiple tabs with different users
- Not supported — single-tab app assumption
- If user logs out in one tab, other tab's next query triggers 401 → redirect to login

### 3. Network error during re-fetch after auth change
- React Query renders error state (red card) for each failed query
- User remains on page with error banners — same as current error handling
- Retry: React Query default retry (3 attempts with exponential backoff)

### 4. Cache clear timing race
- `clear()` + skeleton render must be in same render cycle
- Achieved by `CacheManager` calling `clear()` inside `useEffect` which triggers re-render, and `isPending` going `true` on next render
- **Guaranteed:** No flash of stale data between `clear()` and skeleton display

### 5. Mutation invalidation (D-15)
- After any mutation, `queryClient.invalidateQueries({ queryKey: ['user', userId] })` invalidates all queries for the current user
- Existing data remains visible during background refetch (`isFetching === true`)
- **No skeleton display** — data updates seamlessly when refetch completes
