    Better approach: Check `NODE_ENV` as a simpler gating mechanism:

    ```typescript
    // Production static serving
    if (process.env.NODE_ENV === 'production') {
      // 1. Hashed assets (immutable, cacheable)
      app.use('/assets/*', serveStatic({ root: './frontend/dist' }));
      // 2. SPA fallback — catch all non-API, non-asset routes
      app.get('*', serveStatic({ path: './frontend/dist/index.html' }));
    }
    ```

    This is simpler and doesn't require async at module level. The `NODE_ENV=production` is set in the Dockerfile and docker-compose.

    - The `*` catch-all serves `index.html` for SPA client-side routing

    **# Deployment Guide — FinanceFlow**

    **## Architecture Overview**
    - Single HP T640 Ubuntu homelab
    - Cloudflare Tunnel for HTTPS ingress
    - Docker Compose with 2 services: finance-postgres (PGMQ) + finance-app (Bun + Hono)
    - Diagram showing: Browser → Cloudflare → Cloudflared (host) → Docker app:3000

    **## Prerequisites**
    - Docker 29.4+ and Docker Compose v5.1+
    - Git access to the repository
    - Cloudflared installed on host (for production ingress)

    **## Environment Setup (First Time)**

    1. Clone the repository on the T640 host
    2. Create `.env.production` file from `.env.example`:
       ```
       cp .env.example .env.production
       ```
    3. Fill in all values in `.env.production`:
       - `DATABASE_URL`: Point to the Postgres instance (adjust host if using Docker network)
       - `BETTER_AUTH_SECRET`: Run `openssl rand -hex 32`
       - `BETTER_AUTH_URL`: Your public domain (e.g. `https://finance.yourdomain.com`)
       - `FRONTEND_URL`: Same as BETTER_AUTH_URL
       - `TRUSTED_ORIGINS`: Same as BETTER_AUTH_URL
       - `OPENROUTER_API_KEY`: From OpenRouter dashboard
       - Optional: OAuth client IDs/secrets

    **## First Deployment**

    1. Ensure Postgres container is running (or start it):
       ```
       docker compose up -d db
       ```
    2. Wait for Postgres to be healthy (check with `docker ps`)
    3. Build and start the app:
       ```
       docker compose up -d app
       ```
    4. Check container logs:
       ```
       docker compose logs app
       ```
    5. Verify health:
       ```
       curl http://localhost:3000/health
       ```

    **## Database Migrations**
    - Migrations run automatically on container start via `entrypoint.sh`
    - The baseline migration (001) represents the current schema
    - On an existing database, the first run must use fake mode:
      ```
      docker compose exec app bun run db:migrate:fake
      ```
      (Or run before container start if the entrypoint migration fails)
    - For future schema changes: create `002_xxx.sql`, `003_xxx.sql`, etc. in `src/infrastructure/db/migrations/`
    - Rollback: `docker compose exec app bun run db:migrate:down`

    **## Updating**
    1. Pull latest code: `git pull`
    2. Rebuild and restart: `docker compose up -d --build app`
    3. Check logs: `docker compose logs -f app`
    4. Verify: `curl http://localhost:3000/health`

    **## Rollback**
    1. Run down migration: `docker compose exec app bun run db:migrate:down`
    2. Deploy previous version: `git checkout <previous-tag> && docker compose up -d --build app`

    **## Secret Management**
    - `.env.production` is stored on the T640 host filesystem (outside the repo)
    - Volume-mounted into the container at `/app/.env.production` (read-only)
    - Entrypoint script sources the env file before starting processes
    - The file is listed in `.gitignore` and `.dockerignore` to prevent accidental commits

    **## Container Management**
    - Entrypoint manages 4 processes: API server, insights worker, import worker, static file serving
    - HEALTHCHECK hits `/health` every 30 seconds

    - The container exits if migrations fail (fail-fast)

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Container → Host | `.env.production` crosses from host filesystem into container via volume mount |
| Browser → Hono | HTTP/HTTPS requests from browser to Hono API + static serving |
| Hono → filesystem | `serveStatic` reads files from `frontend/dist/` |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-10 | Information Disclosure | `.env.example` | mitigate | `.env.example` contains placeholders, not real secrets. `.env.production` is in `.gitignore` and `.dockerignore`. |
| T-05-11 | Information Disclosure | `serveStatic` directory traversal | mitigate | Hono's `serveStatic` from `hono/bun` restricts to the `root` directory. Only `frontend/dist/` is served. |
| T-05-12 | Spoofing | Production CORS configuration | mitigate | CORS `origin` set to `FRONTEND_URL` env var (configurable). In production same-origin, CORS is a no-op. Auth routes retain existing CORS middleware. |
| T-05-13 | Spoofing | Session cookie domain mismatch (Pitfall 3) | mitigate | Documented in DEPLOYMENT.md: `BETTER_AUTH_URL` must match the public domain. `FRONTEND_URL` must match. `trustedOrigins` must include the public URL. |
| T-05-14 | Spoofing | Auth guard bypass via direct API access | accept | Already mitigated by existing `requireAuth` middleware on all transaction routes. Frontend guard is UX-only; server enforces auth (per D-11 scope). |
| T-05-SC | Tampering | Node/npm packages during build | mitigate | `bun install --frozen-lockfile` in Docker build ensures locked dependency versions. No new packages added in this plan. |
