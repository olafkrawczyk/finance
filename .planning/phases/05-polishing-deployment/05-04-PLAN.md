  <action>
    This task builds the Docker image without requiring the full docker-compose stack (which needs Postgres). The goal is to verify the build succeeds and the image is functional.

    Step 1 — Build the Docker image:
    ```
    docker compose build
    ```

    If docker compose fails due to missing .env.production (which is expected on dev), create a minimal temporary one:
    ```
    cp .env.example .env.production
    ```
    Then retry build. The `.env.production` is referenced by docker-compose.yml's volume mount.

    Step 2 — Verify the build succeeded:
    ```
    docker images finance-finance-app
    ```
    or
    ```
    docker images | grep finance
    ```

    Step 3 — Start the stack and check health:
    ```
    docker compose up -d
    ```

    Step 4 — Verify container health:
    ```
    docker compose ps
    ```
    Both `db` and `app` should show "Up" or "healthy".

    Step 5 — Check container logs for migration output:
    ```
    docker compose logs app
    ```
    Look for: "Migrations complete" and "All processes started"

    Step 6 — Verify HEALTHCHECK:
    ```
    docker inspect --format='{{.State.Health.Status}}' finance-app
    ```
    Should return "healthy" after the initial health check interval.

    Step 7 — Verify /health endpoint:
    ```
    curl http://localhost:3000/health
    ```
    Should return: `{"data":{"ok":true},"error":null,"meta":null}`

    If the App container exits (restart loop), check logs for migration failure. The entrypoint's `set -e` causes exit if migration fails. This is expected on existing dev databases — run `docker compose exec app bun run db:migrate:fake` to baseline the migration, then restart.

    Note: The health endpoint check confirms that the Bun server is running and responsive inside the container. The migration step may produce warnings if the database already has tables with `CREATE TABLE IF NOT EXISTS` — this is expected and safe.
  </action>

  </acceptance_criteria>

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Docker container | HTTP/HTTPS requests traverse from browser to Hono on port 3000 |
| Auth session → Browser cookie | Better Auth session cookie crosses from server to browser |
| Container → Postgres | App connects to db service via Docker internal network |

### STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-15 | Spoofing | Unauthenticated API access | mitigate | Auth guard (`requireAuth` middleware) returns 401 for unauthenticated requests — verified in V-08 |
| T-05-16 | Tampering | Session cookie | mitigate | Better Auth signs session with `BETTER_AUTH_SECRET`. Cookie is HttpOnly (prevent XSS read). Verified in V-07/V-08. |
| T-05-17 | Information Disclosure | Unauthenticated page content | mitigate | Auth guard in App.tsx shows login page before rendering any authenticated content — verified in V-01 |
| T-05-18 | Denial of Service | Container HEALTHCHECK | accept | HEALTHCHECK only checks API process. Workers may crash independently — container must be restarted manually. Acceptable for single-user app per RESEARCH.md. |
| T-05-19 | Repudiation | No session audit trail | accept | Better Auth sessions are ephemeral. No session logging in scope (deferred to future phase). |
| T-05-SC | Tampering | Docker image supply chain | mitigate | Images built from verified base images: `node:22-bookworm` (official) and `oven/bun:1` (official Docker Hub). No third-party images in app build. |
