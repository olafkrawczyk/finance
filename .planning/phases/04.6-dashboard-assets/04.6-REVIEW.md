## Critical Issues

### CR-01: Fragile error string matching for unique constraint detection

**File:** `src/interface-adapters/api/assets.ts:36`
**File:** `src/interface-adapters/api/assets.ts:62`

**Issue:** Both the POST and PUT route handlers detect duplicate asset names by matching the error message string for substrings `'unique'` or `'duplicate'`:

```typescript
// line 36
const status = message.includes('unique') || message.includes('duplicate') ? 409 : 500;

// line 62 (identical pattern)
const status = message.includes('unique') || message.includes('duplicate') ? 409 : 500;
```

PostgreSQL error messages are locale-dependent. When `lc_messages` is set to a non-English locale (e.g., `pl_PL.UTF-8`, `de_DE.UTF-8`), the error text for a unique constraint violation will not contain `'unique'` or `'duplicate'` — these substrings are only present in English error messages. As a result, every unique constraint violation would be returned as a generic 500 (Internal Server Error) instead of a proper 409 (Conflict).

Additionally, Postgres error message formats can vary between major versions, making string matching doubly unreliable.

**Fix:** Use the Postgres SQLSTATE error code (`23505` for unique violation), which is stable across all locales and versions. Postgres.js exposes the error code in the `code` property:

```typescript
// In the catch handler:
try {
  // ... query
} catch (err) {
  const message = err instanceof Error ? err.message : 'Internal server error';
  // Use err.code (postgres.js exposes SQLSTATE) — 23505 = unique_violation
  const isDuplicate = (err as any)?.code === '23505';
  const status = isDuplicate ? 409 : 500;
  return c.json({ data: null, error: { message }, meta: null }, status);
}
```

Apply this same fix in both POST (line 36) and PUT (line 62) route handlers.

### WR-01: Silent asset fetch failure in Dashboard

**File:** `frontend/src/pages/DashboardPage.tsx:54-57`

**Issue:** When `getAssets()` fails in the dashboard (e.g., network error or server down), the error is only logged to the console:

```typescript
.catch((err) => {
  console.error('Failed to load assets in dashboard:', err);
});
```

The Total Net Value card renders with a sum of `0` (line 197: `{formatPln(totalNetValue)} PLN`), which is indistinguishable from the legitimate scenario where the user has zero assets. This silently misleads the user into thinking they have no assets when, in reality, the data failed to load.

**Fix:** Set an error state when asset fetch fails and display a warning in the Total Net Value tile:

```typescript
const [assetsError, setAssetsError] = useState<string | null>(null);

// In the catch:
.catch((err) => {
  console.error('Failed to load assets in dashboard:', err);
  setAssetsError('Nie udało się załadować aktywów');
});

// In the Total Net Value card, conditionally render:
{assetsError ? (
  <p className="text-amber-400 text-xs mt-2">{assetsError}</p>
) : (
  <p className="text-slate-500 text-xs mt-2">
    Suma wszystkich ręcznie śledzonych aktywów
  </p>
)}
```

### WR-02: `set-cookie` only captures first cookie header in tests

**File:** `tests/assets.test.ts:23-27`

**Issue:** The test suite captures the session cookie using `res.headers.get('set-cookie')`, which only returns the first `Set-Cookie` header from the HTTP response. If Better Auth's `signUpEmail` sets multiple cookies (e.g., a session cookie plus an anti-CSRF or a secure flag cookie), only the first one is captured:

```typescript
const setCookie = res.headers.get('set-cookie');
```

The `Headers.get()` method merges multiple headers of the same name using a comma separator per RFC 7230§3.2.2, but `Set-Cookie` does NOT merge (per RFC 6265§5.1.1, multiple cookies are sent as separate headers). In practice, the Fetch API's `headers.get('set-cookie')` returns only the first value, while `headers.getSetCookie()` (or iterating entries) returns all values.

If the auth library relies on multiple cookies for session validation, subsequent test requests may fail with 401 despite the test expecting successful auth.

**Fix:** Use `getSetCookie()` when available (supported in Bun and modern runtimes) or iterate headers to collect all `Set-Cookie` values:

```typescript
// Option 1: use getSetCookie() if available (Bun supports this)
const allCookies = res.headers.getSetCookie?.() || [res.headers.get('set-cookie')].filter(Boolean);
sessionCookie = allCookies.join('; ');

// Option 2: manual iteration
const cookies: string[] = [];
res.headers.forEach((value, key) => {
  if (key.toLowerCase() === 'set-cookie') cookies.push(value);
});
sessionCookie = cookies.join('; ');
```

### WR-03: Excessive use of `any` types bypassing TypeScript safety

**File:** `frontend/src/pages/AssetsPage.tsx:36`
**File:** `frontend/src/pages/DashboardPage.tsx:27,47,98`

**Issue:** Multiple `.map()` calls use `any` as the element type parameter, discarding compile-time type checking:

**AssetsPage.tsx:36**
```typescript
const parsed = (data ?? []).map((a: any) => ({
  id: a.id,
  name: a.name,
  value: parseFloat(a.value),
}));
```

**DashboardPage.tsx:27**
```typescript
const normalized = rows.map((r: any) => ({
  month: r.month,
  wydatki: parseFloat(r.wydatki),
  // ...
}));
```

Using `any` means a backend API change (e.g., renaming `value` to `current_value`) would not be caught by the TypeScript compiler — it would silently produce `NaN` at runtime.

**Fix:** Type the parameters using the known API response shape:

```typescript
// AssetsPage.tsx — create an API response interface
interface AssetApiResponse {
  id: string;
  name: string;
  value: string; // Postgres numeric returns as string
}

// Then use it:
const parsed = (data ?? []).map((a: AssetApiResponse) => ({
  id: a.id,
  name: a.name,
  value: parseFloat(a.value),
}));
```

Apply the same pattern to the Dashboard data mapping with a `SummaryRowApiResponse` interface.

### IN-02: `CASCADE` on `TRUNCATE` for table without FK references

**File:** `tests/assets.test.ts:10`

**Issue:** `TRUNCATE assets CASCADE` applies cascade behavior unnecessarily — the `assets` table has no foreign key references to any other table in the schema. The `CASCADE` keyword implies cascade-truncation of dependent tables, but since nothing references `assets`, it's a no-op.

```sql
TRUNCATE assets CASCADE;
```

**Fix:** Remove `CASCADE` for clarity:

```sql
TRUNCATE assets;
```

### IN-03: Conflicting `font-semibold` and `font-medium` CSS classes

**File:** `frontend/src/pages/AssetsPage.tsx:166`

**Issue:** The heading element has both `font-semibold` and `font-medium` applied. Since they appear in the same `className` string, the later one in the CSS cascade wins (typically `font-medium`, which is the lower weight). The `font-semibold` is effectively dead code:

```tsx
<h2 className="text-2xl font-semibold text-slate-100 font-medium">
  Zarządzanie aktywami
</h2>
```

**Fix:** Remove `font-medium` since `font-semibold` was likely the intended weight:

```tsx
<h2 className="text-2xl font-semibold text-slate-100">
  Zarządzanie aktywami
</h2>
```
