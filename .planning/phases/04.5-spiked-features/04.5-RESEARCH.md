## Summary

This phase implements transaction CRUD (edit all fields, delete) based on validated spike findings (spikes 002, 003). The core work involves: (1) modifying DB triggers to allow UPDATE/DELETE, (2) adding `updated_at` column, (3) creating three new API endpoints, (4) adding use-case functions, and (5) wiring frontend edit/delete UI.

**Key divergence from spike findings:** The spike recommends re-enqueuing to `analysis_queue` on edit, but the user explicitly chose **not** to re-enqueue (D-04). Delete also includes two additional cleanup steps not in the spike: clearing `import_hash` before deletion (D-05) and removing the deleted transaction ID from `insights.linked_transaction_ids` arrays (D-06).

**Primary recommendation:** Follow Approach C from spike 002 (modify triggers, add `updated_at`), but omit PGMQ re-enqueue on edit per user decision. Add a new `EditTransactionPage`-style component that reuses `AddTransactionPage` form layout with pre-fetched data.

### Pattern 1: API Route Pattern (CRUD endpoints)
**What:** Every new endpoint follows the existing pattern in `ledger.ts`: requireAuth → zValidator → param validation → try/catch → envelope response
**When to use:** All new GET/PUT/DELETE endpoints

```typescript
// Pattern from existing POST /transactions, replicate for PUT/DELETE
ledgerRoutes.put(
  '/:id',
  requireAuth,
  zValidator('json', UpdateTransactionSchema, (result, c) => {
    if (!result.success) {
      return c.json(
        { data: null, error: { message: 'Validation failed', details: result.error.flatten() }, meta: null },
        400
      );
    }
  }),
  async (c) => {
    const rawId = c.req.param('id');
    const parseResult = uuidSchema.safeParse(rawId);
    if (!parseResult.success) {
      return c.json({ data: null, error: { message: 'Invalid transaction id' }, meta: null }, 400);
    }
    try {
      const id = parseResult.data;
      const input = c.req.valid('json');
      const tx = await updateTransaction(id, input);
      return c.json({ data: tx, error: null, meta: null }, 200);
    } catch (err) {
      const message = err instanceof Error ? err.message : 'Internal server error';
      return c.json({ data: null, error: { message }, meta: null }, 500);
    }
  }
);
```

### Pattern 2: Delete Transaction (with hash clearing + insight cleanup)
**What:** Delete must run three SQL operations atomically, wrapped in a transaction
**When to use:** Only `deleteTransaction()` use-case

```typescript
export async function deleteTransaction(id: string): Promise<void> {
  await sql.begin(async (sql) => {
    // Step 1: Clear import_hash so original CSV row can be re-imported (D-05)
    await sql`UPDATE transactions SET import_hash = NULL WHERE id = ${id}`;
    // Step 2: Remove ID from insight linked_transaction_ids (D-06)
    await sql`
      UPDATE insights
      SET linked_transaction_ids = array_remove(linked_transaction_ids, ${id})
      WHERE ${id} = ANY(linked_transaction_ids)
    `;
    // Step 3: Hard delete (D-07)
    await sql`DELETE FROM transactions WHERE id = ${id}`;
  });
}
```

## Common Pitfalls

### Pitfall 1: Delete order-of-operations
**What goes wrong:** Running DELETE before clearing `import_hash` means the row is gone and `import_hash` persists. A subsequent re-import of the same CSV row would collide with a hash from a deleted row.
**Why it happens:** Natural inclination is to `DELETE FROM transactions WHERE id = ${id}` first.
**How to avoid:** Always UPDATE import_hash = NULL first, then DELETE. Use a SQL transaction (`sql.begin`) to ensure atomicity.
**Warning signs:** Any implementation that deletes without first clearing the hash.

### Pitfall 2: Insights cleanup query
**What goes wrong:** The `array_remove` query uses `WHERE ${id} = ANY(linked_transaction_ids)` which is correct for UUID[] arrays, but if Postgres complains about type mismatch, the UUID needs explicit casting.
**Why it happens:** Postgres is strict about UUID vs TEXT in array operations.
**How to avoid:** Use parameterized query (`WHERE ${id} = ANY(linked_transaction_ids)`) which postgres.js handles correctly, or cast: `WHERE ${{raw: `'${id}'::uuid`}} = ANY(linked_transaction_ids)`.
**Warning signs:** Type mismatch errors from the DELETE endpoint.

### Pitfall 3: `updated_at` trigger vs. UPDATE SET clause
**What goes wrong:** The `update_transaction_timestamp()` trigger sets `NEW.updated_at = now()`, which overrides any explicit `updated_at` value in the SET clause.
**Why it happens:** BEFORE UPDATE triggers fire before the row is written, and `RETURN NEW` replaces the row data.
**How to avoid:** Don't set `updated_at` in the use-case SQL — let the trigger always set it. The trigger should use `BEFORE UPDATE` so it overrides any client-provided value.
**Warning signs:** `updated_at` not changing on edit, or error about setting `updated_at` explicitly.

### Pitfall 4: Stale ROADMAP.md and REQUIREMENTS.md
**What goes wrong:** ROADMAP.md mentions adding `reference_id` column (deferred per D-08/D-09). REQUIREMENTS.md still says transactions are immutable (REQ-1.2). After this phase, both documents will be out of date.
**How to avoid:** Update ROADMAP.md to remove the `reference_id` item. Update REQUIREMENTS.md REQ-1.2 to reflect that transactions can now be edited and deleted.
**Warning signs:** N/A — this is documentation staleness, not code breakage.

### Pitfall 5: Existing PATCH /transactions/:id/category becomes redundant
**What goes wrong:** After adding PUT /transactions/:id (which allows updating category_id among other fields), the existing PATCH /transactions/:id/category endpoint becomes a subset. Leaving both is harmless but confusing.
**How to avoid:** Keep the PATCH endpoint for backward compatibility — it doesn't conflict with PUT. Optionally deprecate it in a comment.
**Warning signs:** Two routes doing overlapping things.

### Schema Migration (schema.sql)

```sql
-- 1. Add updated_at column
ALTER TABLE transactions ADD COLUMN updated_at TIMESTAMPTZ;

-- 2. Auto-update updated_at on change (BEFORE UPDATE to override any SET clause)
CREATE OR REPLACE FUNCTION update_transaction_timestamp()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_transactions_updated_at ON transactions;
CREATE TRIGGER trg_transactions_updated_at
  BEFORE UPDATE ON transactions FOR EACH ROW
  EXECUTE FUNCTION update_transaction_timestamp();

-- 3. Replace block_immutable_change — allow all updates
CREATE OR REPLACE FUNCTION block_immutable_change()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  RETURN NEW;  -- Allow all updates
END;
$$;

-- 4. Remove delete block
DROP TRIGGER IF EXISTS trg_transactions_no_delete ON transactions;
DROP FUNCTION IF EXISTS block_delete();
```

### Use-Case Functions (use-cases.ts)

```typescript
export async function getTransaction(id: string): Promise<Transaction | undefined> {
  const [row] = await sql`SELECT * FROM transactions WHERE id = ${id}`;
  return row as Transaction | undefined;
}

export async function updateTransaction(id: string, input: UpdateTransactionInput): Promise<Transaction> {
  const [tx] = await sql`
    UPDATE transactions SET
      account_id = ${input.account_id},
      category_id = ${input.category_id ?? null},
      type = ${input.type},
      amount = ${input.amount},
      description = ${input.description ?? null},
      date = ${input.date},
      transfer_to_account_id = ${input.transfer_to_account_id ?? null}
    WHERE id = ${id}
    RETURNING *
  `;
  if (!tx) throw new Error('Transaction not found');
  return tx as Transaction;
  // NOTE: No PGMQ enqueue per D-04
}

export async function deleteTransaction(id: string): Promise<void> {
  await sql.begin(async (sql) => {
    await sql`UPDATE transactions SET import_hash = NULL WHERE id = ${id}`;
    await sql`
      UPDATE insights
      SET linked_transaction_ids = array_remove(linked_transaction_ids, ${id})
      WHERE ${id} = ANY(linked_transaction_ids)
    `;
    await sql`DELETE FROM transactions WHERE id = ${id}`;
  });
}
```

### Frontend API Client (api.ts)

```typescript
export async function getTransaction(id: string) {
  const res = await fetch(`/transactions/${id}`, { credentials: 'include' });
  if (!res.ok) {
    const errorJson = await res.json().catch(() => ({}));
    throw new Error(errorJson.error?.message || `Failed to fetch transaction: ${res.statusText}`);
  }
  const json = await res.json();
  return json.data;
}

export async function updateTransaction(id: string, data: {
  account_id: string;
  category_id?: string | null;
  type: 'income' | 'expense' | 'transfer';
  amount: string;
  description?: string | null;
  date: string;
  transfer_to_account_id?: string | null;
}) {
  const res = await fetch(`/transactions/${id}`, {
    method: 'PUT',
    credentials: 'include',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data),
  });
  if (!res.ok) {
    const errorJson = await res.json().catch(() => ({}));
    throw new Error(errorJson.error?.message || `Failed to update transaction: ${res.statusText}`);
  }
  const json = await res.json();
  return json.data;
}

export async function deleteTransaction(id: string) {
  const res = await fetch(`/transactions/${id}`, {
    method: 'DELETE',
    credentials: 'include',
  });
  if (!res.ok) {
    const errorJson = await res.json().catch(() => ({}));
    throw new Error(errorJson.error?.message || `Failed to delete transaction: ${res.statusText}`);
  }
  const json = await res.json();
  return json.data;
}
```
