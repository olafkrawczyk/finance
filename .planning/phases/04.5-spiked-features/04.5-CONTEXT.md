<decisions>
## Implementation Decisions

### Edit Transaction
- **D-01:** All fields are editable: type, amount, date, description, account_id, category_id, transfer_to_account_id
- **D-02:** Edit form reuses the existing AddTransactionPage pattern — navigate to page with pre-filled data, not a modal
- **D-03:** Existing category assignment is preserved on edit — not cleared
- **D-04:** Edit does NOT re-enqueue to PGMQ `analysis_queue` — skip re-analysis on edit

### Delete Transaction
- **D-05:** On delete, set `import_hash = NULL` before deleting so the same CSV row can be re-imported
- **D-06:** On delete, also remove the deleted transaction ID from `insights.linked_transaction_ids` arrays
- **D-07:** Hard delete (no soft delete / deleted_at column)

### Import Dedup
- **D-08:** No changes to the dedup mechanism — current `date|amount|description` hash is sufficient
- **D-09:** No `reference_id` column added. No LLM prompt changes for extraction

### the agent's Discretion
- API endpoint design (PUT/DELETE/GET single transaction) — follow existing ledger route patterns in `src/interface-adapters/api/ledger.ts`
- Validation schema for update — reuse `CreateTransactionSchema` as `UpdateTransactionSchema`
- Trigger modification SQL — follow Approach C from spike 002
- Frontend details: navigate to edit page, confirmation dialog for delete, toast notifications

</decisions>
