## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (page title) | 24px (`text-2xl`) | 600 (`font-semibold`) | 1.25 (`leading-tight`) |
| Heading (section title, card title, modal title) | 18px (`text-lg`) | 600 (`font-semibold`) | 1.25 (`leading-tight`) |
| Subheading (section labels, uppercase headings) | 12px (`text-xs`) | 600 (`font-semibold`) | 1.25 (`leading-tight`) |
| Body (content, descriptions) | 14px (`text-sm`) | 400 (`font-normal`) | 1.5 (`leading-relaxed`) |
| Label (form labels, table headers) | 12px (`text-xs`) | 600 (`font-semibold`) | 1.25 (`leading-tight`) |
| UI (buttons, nav, table cells) | 14px (`text-sm`) | 600 (`font-semibold`) or 500 (`font-medium`) | 1.4 |
| Metric (numbers, amounts) | 14px (`text-sm`) or 24px (`text-2xl`) | 700 (`font-bold`) or 600 (`font-semibold`) | 1.25 (`font-mono` for alignment) |

**Line height defaults:**
- Body / paragraph text: `leading-relaxed` (1.5)
- Headings: `leading-tight` (1.25)
- Labels / metadata: `leading-tight` (1.25)
- Monospace numbers: `font-mono` (tabular figures, consistent width) — used in MonthSidebar opening balance display

**Usage constraint:** Maximum 2 weights per element — `font-semibold` (600) for headings/buttons, `font-bold` (700) for emphasis on CTAs and numerical values. `font-medium` (500) appears in existing code for nav items and table cells — keep as-is. Do **not** introduce `font-light` (300) or `font-extrabold` (800).

**Pre-populated from:** Phase 10 UI-SPEC typography table, confirmed via codebase scan.

**Modal structure:**
- Overlay: `fixed inset-0 bg-slate-950/80 backdrop-blur-sm z-50 flex items-center justify-center p-4`
- Container: `bg-slate-900 border border-slate-800 rounded-xl p-8 max-w-md w-full shadow-2xl space-y-6`
- Warning box: `bg-red-950/50 border border-red-800 text-slate-100 rounded-lg p-4 flex gap-3 items-start`
- Input: `w-full bg-slate-950 border border-slate-800 rounded px-3 py-2 text-slate-100 outline-none focus:border-blue-500`

- Input must exactly match `DELETE {accountName}` (e.g., "DELETE ING Business")
- Matching is case-sensitive for the account name portion — matches the actual stored name
- Delete button enables only when input matches the challenge phrase exactly
- On confirm: calls `onConfirm()`, closes modal, resets input
- On cancel: calls `onClose()`, closes modal, resets input
- Keyboard: Enter triggers confirm (if enabled), Escape triggers cancel

**Note:** This is a new component distinct from `DestructiveConfirmModal.tsx` (which uses a static `MIGRATE` challenge). The challenge phrase is **dynamic** — it incorporates the target account name.

## Interaction Contract

### 1. Page Load (AccountPage)

| State | Visual | Condition |
|-------|--------|-----------|
| Loading skeleton | 2-column layout: left column has skeleton h-8 "Zarządzanie kontami" + 3 skeleton rows; right column has skeleton form card | `isPending === true` on `useAccounts()` |
| Error | Red banner card with error message | `isError === true` |
| Empty | "Brak zapisanych kont" centered in card | `data` is empty array |
| Data loaded | Table with rows + action buttons | `data` populated |

### 2. Account Creation Flow

| Step | Visual | State |
|------|--------|-------|
| Idle | Empty form fields ready | Pre-submit |
| Field validation | Inline error via `setError()` red banner | Missing/invalid fields |
| Submitting | Button shows "Dodawanie...", input `disabled` | `submitting === true` |
| Success | Green banner "Pomyślnie dodano nowe konto", form resets, list refetches | Mutation success |
| Failure | Red banner with error message | Mutation error |
| Duplicate name | Red banner "Konto o tej nazwie już istnieje" | 409 from server |

### 3. Inline Edit Flow

| Step | Visual | State |
|------|--------|-------|
| Idle | Row displays data spans | `editingId === null` |
| Edit mode | Fields become editable inputs, buttons show [Zapisz] [Anuluj] | `editingId === row.id` |
| Saving | [Zapisz] shows "Zapisywanie...", disabled | `savingId === row.id` |
| Success | Exit edit mode, show green banner, refetch list | Mutation success |
| Failure | Stay in edit mode, show red banner | Mutation error |
| Cancel | Exit edit mode, discard changes | `handleCancelEdit()` called |

### 4. Delete Flow (typed confirmation)

| Step | Visual | State |
|------|--------|-------|
| Trigger | User clicks "Usuń" button → open modal | `deleteModalOpen === true` |
| Modal appears | Backdrop overlay + modal with challenge input | `isOpen === true` |
| Input empty | Delete button disabled (opacity-50, cursor-not-allowed) | Input doesn't match challenge |
| Input matching | Delete button enables (red-800 hover:red-700) | Input matches `DELETE {name}` |
| Confirming | Button shows "Usuwanie..." (or debounce) | `deleting === true` |
| Success | Close modal, green banner "Pomyślnie usunięto konto", refetch list | Delete mutation success |
| Failure (has transactions) | Close modal, red banner "Nie można usunąć konta, ponieważ ma {X} transakcji" | 409 from server |
| Failure (generic) | Close modal, red banner with error message | Other error |
| Cancel | Close modal, reset input | `onClose()` called |

### 5. Mutation Invalidation

After any account CRUD mutation (create, update, delete):
```typescript
queryClient.invalidateQueries({ queryKey: ['user', userId] });
```
This follows the Phase 10 pattern (D-15) — broad invalidation triggers background refetch for all queries. Existing data remains visible during refetch (`isFetching === true`). **No skeleton display** during background refetch.

### 6. Combined Net Worth Line Interaction

- Default state: Both "Stan konta" (blue) and "Wartość netto" (purple) lines visible
- Toggle click: Hides/shows the purple net worth line (just visual toggle, no data re-fetch)
- Toggle state remembered via React `useState` local to `DashboardPage`
- Tooltip: Shows both values for hovered month

### 7. Asset Value Change → Snapshot (D-10)

- User edits asset value in AssetsPage (existing inline edit)
- On successful `updateAsset` mutation: backend auto-creates a snapshot of the old value
- **No visible UX change** for the snapshot — it's transparent to the user
- The net worth chart automatically reflects the new value on next refetch

### 8. Balance Chart Data Update (D-04, D-11)

- `getMonthlySummary` backend computes:
  - `stan_konta`: per-account starting balance aggregation + transaction running balance
  - `wartosc_netto`: `stan_konta` + forward-filled asset values from snapshots
- Frontend receives both values per month row
- BalanceChart renders both lines
- No new API calls needed — `useMonthlySummary()` already supplies the data

## Interaction Edge Cases

### 1. Delete account with many transactions
- Backend returns 409 with count of referencing transactions
- UI shows: "Nie można usunąć konta, ponieważ ma {X} transakcji" (red banner)
- Modal closes first (error is non-modal banner), user sees which account failed

### 2. Duplicate account name
- Backend returns 409 from `UNIQUE(user_id, name)` constraint
- UI shows: "Konto o tej nazwie już istnieje" (red banner in form area)
- User must rename the account

### 3. All accounts deleted
- If user deletes all accounts, empty state shows "Brak zapisanych kont"
- User can still create new accounts via the right panel form
- Signup still creates 2 default accounts (D-07), so this is rare for established users

### 4. Edit while another user modifies same account
- Not handled — single-user-per-instance app. If concurrent edit occurs via API, last write wins.

### 5. Very long account name
- Table: `overflow-x-auto` on table container — matches existing pattern in AssetsPage
- Name column: `max-w-[200px] truncate` or just let the table scroll
- Modal: Challenge input must accept the full name — width accommodates

### 6. Typed delete confirmation with special characters
- Input comparison: exact string match (no trimming needed for `DELETE ` prefix)
- Account names like "ING (Business)" work correctly with exact match

### 7. Null starting_balance_date
- Display: "—" (em dash) in the date column cell
- Internally: `starting_balance_date=NULL` means starting_balance is treated as effective from the earliest transaction month (D-05)
- Edit mode: date input shows empty if null, user can set it

### 8. All starting_balances at 0
- No backward-compatibility issue (D-04): falls back to `monthly_opening_balances` for legacy behavior
- UI displays 0 balances correctly

### 9. Asset value update → snapshot race condition
- Snapshot creation is synchronous and happens before the update response
- No loading state needed — backend handles it atomically
- Failure to create snapshot: backend should roll back the asset update (or handle gracefully)
- UI shows the asset update success; if snapshot fails, the asset value is still updated
