    The test must cover SCOPE-01 through SCOPE-06 with a matrix: 2 users × 6 resource types × CREATE/READ/UPDATE/DELETE/LIST with negative cross-user access tests.

    **Structure:**

    ```typescript
    import { describe, it, expect, beforeAll } from 'bun:test';
    import { app } from '../index';
    import { auth } from '../src/auth';
    import sql from '../src/infrastructure/db/client';

    const USER_A_EMAIL = 'scope-user-a@test.com';
    const USER_B_EMAIL = 'scope-user-b@test.com';
    const USER_A_PASSWORD = 'testpass123';
    const USER_B_PASSWORD = 'testpass456';

    let userASession: string;
    let userBSession: string;
    let userAAccountId: string;
    let userBCategoryId: string;
    let userATransactionId: string;
    let userBOpeningBalanceId: string;
    let userAAssetId: string;
    let userBAccountId: string;
    ```

    **Setup (beforeAll):**
    1. Clean up any existing test users from prior runs (`DELETE FROM "user" WHERE email IN (...)` via SQL)
    2. Create User A via `auth.api.signUpEmail({ body: { email: USER_A_EMAIL, password: USER_A_PASSWORD, name: 'Scope User A' }, asResponse: true })` — this triggers the signup hook, creating default categories + accounts
    3. Create User B via same method
    4. Extract session cookies from responses
    5. Fetch User A's account ID (the first one from the signup hook defaults) and User B's category ID
    6. Create a transaction for User A and an opening balance for User B for use in negative tests

    **Test groups:**

    *Group 1: User A creates resources (SCOPE-02)*
    - Create transaction for User A (POST /transactions) → 201, verify user_id via SQL
    - Create asset for User A (POST /assets) → 201
    - Create opening balance for User A (POST /opening-balance) → 201

    *Group 2: User A reads own resources (SCOPE-01)*
    - GET /transactions → returns User A's transactions
    - GET /assets → returns User A's assets
    - GET /opening-balance → returns User A's balances
    - GET /accounts → returns User A's accounts
    - GET /categories → returns User A's categories

    *Group 3: User B cannot access User A's resources (SCOPE-01/SCOPE-05 — 404, not 403)*
    - GET /transactions with User A's transaction ID + User B's cookie → 404
    - PUT /transactions/:id with User A's transaction ID + User B's cookie → 404
    - DELETE /transactions/:id with User A's transaction ID + User B's cookie → 404
    - PATCH /transactions/:id/category with User A's transaction ID + User B's cookie → 404
    - GET /assets/:id with User A's asset ID + User B's cookie → 404
    - PUT /assets/:id with User A's asset ID + User B's cookie → 404
    - DELETE /assets/:id with User A's asset ID + User B's cookie → 404
    - GET /opening-balance with User B's balance ID + User A's cookie → 404
    - PUT /opening-balance/:id with User B's balance ID + User A's cookie → 404

    *Group 4: User B operates within own data (positive isolation)*
    - User B creates own transaction → 201
    - User B lists own transactions → only User B's transactions returned
    - User B lists own accounts → only User B's accounts returned
    - User B lists own categories → 25 categories (from signup hook)

    *Group 5: Auth-required routes reject unauthenticated (SCOPE-06)*
    - POST /transactions without cookie → 401
    - GET /transactions without cookie → 401
    - POST /assets without cookie → 401
    - POST /import without cookie → 401
    - POST /opening-balance without cookie → 401

    **Cleanup (afterAll):**
    ```typescript
    afterAll(async () => {
      await sql`DELETE FROM "user" WHERE email IN (${USER_A_EMAIL}, ${USER_B_EMAIL})`;
    });
    ```

    **Important test patterns:**
    - For 404 tests on resources that are accessed by ID (GET /:id, PUT /:id, DELETE /:id): the actual response code depends on how the route is implemented. If the route does a SELECT first to verify existence and returns 404 if not found, that's correct. If it attempts the operation and returns 404 based on affected rows, also correct. The key test is that a 404 is returned rather than the data or a success.
    - Use `app.request(url, { method, headers: { Cookie: sessionCookie }, body })` pattern from api.test.ts
    - Verify response status with `expect(res.status).toBe(404)` for cross-user access
    - For POST bodies, use `JSON.stringify()` with `Content-Type: application/json`

    **What to avoid:**
    - Do NOT test worker isolation — that's Phase 8 scope (WORKER-01 through WORKER-04)
    - Do NOT test migration TRUNCATE behavior — that's an admin operation
    - Do NOT test insights module — it's already scoped and tested separately
    - Do NOT hardcode UUIDs — fetch IDs dynamically from SQL queries in setup

    ```typescript
    import { describe, it, expect, beforeAll, afterAll } from 'bun:test';
    import { app } from '../index';
    import { auth } from '../src/auth';
    import sql from '../src/infrastructure/db/client';

    let sessionCookie: string;

    ```

    **Setup (beforeAll):**
    1. Clean up: `await sql`DELETE FROM "user" WHERE email = ${TEST_EMAIL}``
    2. Sign up new user via `auth.api.signUpEmail({...})` — this triggers the signup hook
    3. Extract session cookie from response
    4. Get user ID: extract from session or query `SELECT id FROM "user" WHERE email = ${TEST_EMAIL}`

    **Tests:**

    *Group 1: SEED-01 — Default categories*
    ```typescript
    it('creates 25 default categories for new user', async () => {
      const rows = await sql`
        SELECT * FROM categories WHERE user_id = ${userId}
      `;
      expect(rows.length).toBe(25);
    });

    it('default categories have llm_description populated', async () => {
      const rows = await sql`
        SELECT * FROM categories WHERE user_id = ${userId} AND llm_description IS NULL
      `;
      expect(rows.length).toBe(0);
    });

    it('default categories include known names (biedronka, ZUS, etc.)', async () => {
      const rows = await sql`
        SELECT name FROM categories WHERE user_id = ${userId} ORDER BY name
      `;
      const names = rows.map(r => r.name);
      expect(names).toContain('biedronka');
      expect(names).toContain('ZUS');
      expect(names).toContain('paliwo');
    });

    it('default categories are accessible via GET /categories', async () => {
      const res = await app.request('/categories', {
        headers: { Cookie: sessionCookie },
      });
      expect(res.status).toBe(200);
      const json = await res.json();
      expect(json.data.length).toBe(25);
    });
    ```

    *Group 2: SEED-02 — Default accounts*
    ```typescript
    it('creates 2 default accounts for new user', async () => {
      const rows = await sql`
        SELECT * FROM accounts WHERE user_id = ${userId}
      `;
      expect(rows.length).toBe(2);
    });

    it('default accounts include ING Business and IPKO Personal', async () => {
      const rows = await sql`
        SELECT name FROM accounts WHERE user_id = ${userId} ORDER BY name
      `;
      const names = rows.map(r => r.name);
      expect(names).toContain('ING Business');
      expect(names).toContain('IPKO Personal');
    });

    it('default accounts are accessible via GET /accounts', async () => {
      const res = await app.request('/accounts', {
        headers: { Cookie: sessionCookie },
      });
      expect(res.status).toBe(200);
      const json = await res.json();
      expect(json.data.length).toBe(2);
    });
    ```

    ```typescript
    it('seeding happens exactly once — second user gets fresh defaults', async () => {
      const email2 = 'seeding-test-2@example.com';
      await sql`DELETE FROM "user" WHERE email = ${email2}`;

      const res2 = await auth.api.signUpEmail({
        body: { email: email2, password: 'testpass456', name: 'Seeding Test 2' },

      `;
      expect(rows.length).toBe(25);

      // Cleanup
      await sql`DELETE FROM "user" WHERE email = ${email2}`;
    });
    ```
